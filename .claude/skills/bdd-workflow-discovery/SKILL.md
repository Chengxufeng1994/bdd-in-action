# BDD 工作流程發現技能

## 概述

此技能引導使用者完成 BDD 的前期工作流程：

1. **功能描述階段**：協助撰寫清晰的使用者故事
2. **故事評估與拆分階段**：評估故事大小，必要時應用拆分模式
3. **Example Mapping 階段**：為每個小故事發現業務規則和具體測試範例

## 使用方式

```bash
# 啟動 BDD 工作流程發現
/bdd-workflow-discovery

# 或直接提供功能描述
/bdd-workflow-discovery "我想要實現銀行轉帳功能"
```

## 工作流程

### 階段一：功能描述

基於 `docs/chapter03/04-feature-description.md` 的指導原則，協助使用者：

1. **識別利益相關者**
   - 誰會使用這個功能？
   - 誰會從中受益？

2. **定義功能需求**
   - 具體需要什麼功能？
   - 功能的範圍是什麼？

3. **闡明業務價值**
   - 為什麼需要這個功能？
   - 解決什麼問題？
   - 創造什麼價值？

4. **選擇故事格式**
   - 格式一（目標導向）：`In order to... As a... I want...`
   - 格式二（角色導向）：`As a... I want... So that...`

5. **驗證 INVEST 原則**
   - Independent（獨立的）
   - Negotiable（可協商的）
   - Valuable（有價值的）
   - Estimable（可估算的）
   - Small（小的）
   - Testable（可測試的）

### 階段二：故事評估與拆分

基於 `docs/chapter03/06-story-splitting.md` 的指導原則，評估並拆分故事：

1. **評估故事大小**
   - 檢查是否符合 INVEST 的 S（Small）原則
   - 識別故事太大的徵兆：
     - 無法估算工作量
     - 超過一個 Sprint
     - Example Mapping 會超過 30 分鐘
     - 規則數量可能 > 8 個

2. **選擇拆分模式**（如果需要拆分）
   - 工作流程步驟（Workflow Steps）
   - 業務規則變化（Business Rule Variations）
   - 簡單/複雜（Simple/Complex）
   - 資料變化（Variations in Data）
   - CRUD 操作（Create/Read/Update/Delete）
   - 介面（Interface）
   - 效能/規模（Performance/Scale）

3. **應用拆分模式**
   - 確保垂直切分（每個故事包含完整功能切片）
   - 每個小故事都有獨立的業務價值
   - 避免水平切分（技術層次分離）

4. **驗證拆分結果**
   - 每個小故事符合 INVEST 原則
   - 可在 1-2 天內完成
   - 依賴關係最小化

5. **排序優先級**
   - 使用 MoSCoW 方法（Must/Should/Could/Won't）
   - 識別 MVP（最小可行產品）

### 階段三：Example Mapping

基於 `docs/chapter03/05-example-mapping.md` 的方法，引導使用者：

1. **識別業務規則**（藍色便利貼 📏）
   - "在什麼情況下..."
   - "如果...那麼..."
   - "只有當..."

2. **創建具體範例**（綠色便利貼 📝）
   - 為每個規則提供至少 2 個範例
   - 涵蓋正常情況和異常情況
   - 使用 Given-When-Then 格式

3. **記錄待解決問題**（紅色便利貼 ❓）
   - 業務規則不明確的地方
   - 技術可行性問題
   - 整合問題

4. **評估準備度**
   - 綠燈：準備好開發
   - 黃燈：需要更多工作
   - 紅燈：故事太大，需拆分

## 實現邏輯

### 步驟 1：收集功能描述

```
如果用戶沒有提供描述：
  詢問：「請描述你想要實現的功能」

接收用戶輸入後：
  1. 總結理解
  2. 確認理解正確
```

### 步驟 2：引導撰寫使用者故事

```
詢問以下問題：

1. 利益相關者（As a）
   Q: "誰會使用這個功能？"
   Q: "他們的角色是什麼？"

2. 功能需求（I want）
   Q: "具體需要什麼功能或能力？"
   Q: "功能的核心是什麼？"

3. 業務價值（So that）
   Q: "為什麼需要這個功能？"
   Q: "它解決了什麼問題？"
   Q: "創造了什麼價值？"

基於回答，提供兩種格式的使用者故事：
- 格式一（目標導向）
- 格式二（角色導向）

詢問用戶偏好哪種格式
```

### 步驟 3：驗證 INVEST 原則

```
檢查使用者故事是否符合 INVEST：

如果故事太大（S - Small）：
  → 進入步驟 3.5（故事拆分）

如果缺乏明確價值（V - Valuable）：
  引導重新思考業務價值

如果無法測試（T - Testable）：
  協助定義驗收標準
```

### 步驟 3.5：故事拆分流程（如果需要）

```
說明拆分必要性：
"這個故事看起來比較大，讓我們將它拆分成更小的故事，
這樣每個故事都可以在 1-2 天內完成。"

階段 3.5.1：識別拆分模式
--------------------------------------
詢問引導問題：

Q1: "這個功能有明確的工作流程步驟嗎？"
    → 適用「工作流程步驟」模式

Q2: "不同的業務規則或條件會導致不同行為嗎？"
    → 適用「業務規則變化」模式

Q3: "是否有簡單版本和複雜版本的區別？"
    → 適用「簡單/複雜」模式

Q4: "是否涉及不同類型的資料或介面？"
    → 適用「資料變化」或「介面」模式

Q5: "是否是標準的 CRUD 操作？"
    → 適用「CRUD 操作」模式

階段 3.5.2：應用拆分模式
--------------------------------------
根據選擇的模式，引導拆分：

工作流程步驟：
  "讓我們將這個流程分解成步驟：
   第一步是什麼？第二步？...
   每個步驟都可以是一個獨立的故事。"

業務規則變化：
  "有哪些不同的情況或類型？
   我們可以為每種情況創建一個故事。"

簡單/複雜：
  "什麼是最基本的版本（MVP）？
   然後我們可以逐步增加複雜度。"

確保垂直切分：
  "每個小故事都要包含完整的功能切片
   （資料庫 + API + 介面 + 測試）"

階段 3.5.3：排序優先級
--------------------------------------
使用 MoSCoW 方法：

Q: "哪些故事是 Must have（必須有的）？"
   → 標記為 Sprint 1 / MVP

Q: "哪些故事是 Should have（應該有的）？"
   → 標記為 Sprint 2

Q: "哪些故事是 Could have（可以有的）？"
   → 標記為 Sprint 3

Q: "哪些故事是 Won't have（暫不做的）？"
   → 記錄到 Backlog

階段 3.5.4：驗證拆分結果
--------------------------------------
對每個小故事檢查：

□ 獨立的（不依賴其他故事）
□ 有業務價值（可獨立展示）
□ 可估算的（1-2 天）
□ 可測試的（清晰驗收標準）

如果都符合 → 拆分成功
如果不符合 → 重新調整

輸出格式：
展示拆分後的故事列表：
```
📋 拆分後的故事

Story 1.1：[名稱]
As a [角色]
I want [需求]
So that [價值]
優先級：Must have
估算：2 天

Story 1.2：[名稱]
...
```
```

### 步驟 4：Example Mapping 會議（為每個小故事）

```
說明 Example Mapping 流程：
"現在我們將進行 Example Mapping 來發現業務規則和範例。
我會扮演主持人角色，引導你完成這個過程。"

階段 2.1：探索業務規則（10 分鐘）
--------------------------------------
詢問引導問題：

Q1: "在什麼情況下這個功能會成功？"
    → 記錄為規則（📏）

Q2: "有哪些限制或約束？"
    → 記錄為規則（📏）

Q3: "什麼時候會失敗？"
    → 記錄為規則（📏）

Q4: "不同的使用者角色規則一樣嗎？"
    → 可能需要更多規則

對每個規則：
  展示格式：📏 規則 X：[規則描述]
  詢問："這個規則總是適用嗎？有例外嗎？"

階段 2.2：創建具體範例（10 分鐘）
--------------------------------------
對每個規則：

Q: "請提供一個這個規則的具體例子"
   → 記錄為範例（📝）

引導使用 Given-When-Then 格式：
  Given [初始狀態]
  When [執行動作]
  Then [預期結果]

確保涵蓋：
  ✓ 正常情況（快樂路徑）
  ✓ 異常情況（錯誤處理）
  ✓ 邊界情況（極限值）

階段 2.3：識別問題（5 分鐘）
--------------------------------------
詢問：

Q1: "有哪些地方還不清楚？"
    → 記錄為問題（❓）

Q2: "需要與誰確認？"
    → 標記負責人

Q3: "有技術可行性的疑慮嗎？"
    → 記錄為技術問題

Q4: "需要與其他系統整合嗎？"
    → 記錄為整合問題
```

```
### 文檔儲存

**目錄結構：**
```
項目根目錄/
└── specs/
    ├── [功能名稱]-discovery.md       # 完整發現文檔（包含故事拆分和Example Mapping）
    └── ...
```

**檔案命名規則：**
- 功能名稱使用小寫和連字號（kebab-case）
- 範例：`user-login-discovery.md`、`bank-transfer-discovery.md`

**儲存邏輯：**
1. 確保 `specs/` 目錄存在（使用 `mkdir -p specs`）
2. 生成完整的 Markdown 文檔
3. 使用 Write 工具儲存文件到 `specs/[功能名稱]-discovery.md`
4. 向用戶確認文件路徑

**設計理念：**
- `specs/` 符合 BDD 的「Specification by Example」概念
- 單一文檔包含完整的發現過程（原始故事 + 拆分 + Example Mapping）
- 便於追溯和審查整個需求發現過程
```

## 輸出範例

### 範例 1：銀行轉帳功能

```markdown
# 功能：銀行帳戶間轉帳

## 使用者故事

**As a** 銀行客戶
**I want** 在我的帳戶間轉移資金
**So that** 我可以靈活管理我的財務

## Example Mapping 結果

### 📏 業務規則

#### 規則 1：餘額驗證

來源帳戶必須有足夠的餘額來完成轉帳。

📝 **範例 1.1**：餘額充足
\`\`\`
Given 支票帳戶有 $1,000
And 儲蓄帳戶有 $500
When 從支票轉 $200 到儲蓄
Then 支票帳戶剩 $800
And 儲蓄帳戶變成 $700
\`\`\`

📝 **範例 1.2**：餘額不足
\`\`\`
Given 支票帳戶有 $100
When 嘗試轉 $200 到儲蓄
Then 轉帳失敗
And 顯示「餘額不足」錯誤
\`\`\`

#### 規則 2：轉帳限額

單筆轉帳不得超過 $10,000

📝 **範例 2.1**：在限額內
\`\`\`
Given 支票帳戶有 $20,000
When 轉 $9,999 到儲蓄
Then 轉帳成功
\`\`\`

📝 **範例 2.2**：超過限額
\`\`\`
Given 支票帳戶有 $20,000
When 嘗試轉 $10,001
Then 轉帳失敗
And 顯示「超過單筆限額」錯誤
\`\`\`

### ❓ 待解決問題

1. VIP 客戶的限額是否不同？ - 負責人：產品經理
2. 轉帳是即時處理還是有延遲？ - 負責人：技術架構師
3. 是否需要雙重認證？ - 負責人：安全團隊

## 評估

- 規則數量：2
- 範例數量：4
- 問題數量：3
- 準備度：🟡 黃燈（需要解決問題後再開發）

## 下一步

1. ✅ 安排會議與產品經理確認 VIP 限額（本週內）
2. ✅ 技術架構師提供轉帳處理方案（2 天內）
3. ✅ 安全團隊評估認證需求（1 週內）
4. ⏭️ 問題解決後，編寫 Gherkin 場景
5. ⏭️ 開始開發和測試
```

### 漸進式引導

```
第一輪：基本理解
  → 功能是什麼？

第二輪：深入探索
  → 誰、什麼、為什麼？

第三輪：規則發現
  → 什麼時候成功？失敗？

第四輪：範例創建
  → 給我一個具體例子

第五輪：問題識別
  → 還有什麼不清楚？
```

## 最佳實踐

### ✅ Do（應該做）

1. **遵循 DFS 原則** ⭐
   - 一次專注一個故事
   - 完整探索當前故事的所有規則和範例
   - 完成後再進入下一個故事
   - 從優先級最高的故事開始（Must have）

2. **保持互動**
   - 每次只問 1-2 個問題
   - 根據回答調整下一個問題
   - 總結理解並確認

3. **使用具體範例**
   - 要求提供實際數據
   - 避免抽象描述
   - 使用 Given-When-Then

4. **記錄所有內容**
   - 規則、範例、問題都要記錄
   - 使用清晰的格式
   - 視覺化組織（emoji）

5. **驗證理解**
   - 重複使用者的話
   - 確認理解正確
   - 澄清模糊之處

6. **評估準備度**
   - 根據規則、範例、問題數量評估
   - 提供明確的下一步建議
   - 識別是否需要拆分故事

### ❌ Don't（避免做）

1. **不要使用 BFS（廣度優先）** ⭐
   - 不要同時探索多個故事
   - 不要淺層掃過所有規則
   - 不要等所有故事都討論完才開始開發
   - 遵循 DFS 原則：深入一個，完成一個

2. **不要一次問太多**
   - 避免長問卷
   - 分階段進行
   - 保持對話流暢

3. **不要假設**
   - 不要猜測業務規則
   - 總是確認理解
   - 記錄不確定的地方

4. **不要跳過範例**
   - 規則必須有範例支持
   - 至少 2 個範例（正常+異常）
   - 範例要具體

5. **不要忽略問題**
   - 所有問題都要記錄
   - 指定負責人
   - 設定解決期限

## 🎯 DFS 原則（深度優先探索）

**核心理念：一次專注一個故事，完整探索後再進入下一個**

### 為什麼使用 DFS？

```
❌ BFS (廣度優先) - 避免這樣做：
───────────────────────────────
1. 列出所有拆分後的故事
2. 為所有故事做淺層 Example Mapping
3. 一起討論所有規則
→ 結果：信息過載、細節遺失、無法立即開始開發

✅ DFS (深度優先) - 應該這樣做：
───────────────────────────────
1. 選擇第一個故事（通常是 MVP 中的 Story 1.1）
2. 深入進行完整的 Example Mapping
   - 探索所有業務規則
   - 創建所有相關範例
   - 識別所有待解決問題
3. 評估準備度
4. 如果準備好 → 可以立即開始開發
5. 重複步驟 1-4 處理下一個故事

→ 結果：每個故事都有完整細節、可以立即開發、持續交付價值
```

### DFS 工作流程

```
Story 1.1 (Must have)
│
├─ Example Mapping (深入)
│  ├─ 規則 1 → 範例 1.1, 1.2
│  ├─ 規則 2 → 範例 2.1, 2.2
│  └─ 問題識別 → 全部記錄
│
├─ 準備度評估
│  └─ 綠燈 ✅
│
└─ 可以開始開發！
    │
    ├─ 編寫 Gherkin
    ├─ 實現步驟定義
    └─ 開發完成
        │
        └─ 進入下一個故事
            │
            Story 1.2 (Must have)
            │
            ├─ Example Mapping (深入)
            └─ ...
```

### 實踐指南

**步驟 1：選擇當前故事**
- 從優先級最高的開始（Must have）
- 通常是 Story 1.1

**步驟 2：深入 Example Mapping**
- 不要急於進入下一個故事
- 充分探索所有規則和範例
- 記錄所有疑問和問題

**步驟 3：完成當前故事後再繼續**
- 確保準備度為綠燈
- 或者問題已記錄並指派負責人
- 然後才進入下一個故事

**步驟 4：迭代進行**
- Story 1.1 完成 → Story 1.2
- Story 1.2 完成 → Story 1.3
- 每個故事都經過完整的 DFS

### 優勢對比

| 特性 | DFS (深度優先) | BFS (廣度優先) |
|------|----------------|----------------|
| 專注度 | 高 - 一次一個故事 | 低 - 同時處理多個 |
| 信息保留 | 完整 - 深入細節 | 部分 - 淺層概覽 |
| 可開發性 | 立即 - 完成即可開發 | 延遲 - 需等所有完成 |
| 風險控制 | 早期發現問題 | 後期集中暴露 |
| 交付節奏 | 持續交付 | 大批次交付 |
| 認知負荷 | 低 - 逐個處理 | 高 - 同時追蹤 |

### 範例對話

```
❌ 錯誤做法（BFS）：
───────────────────
Claude: "我們已經拆分成 5 個故事，現在我們快速過一遍每個故事的業務規則..."
→ 結果：每個故事都只有淺層理解

✅ 正確做法（DFS）：
───────────────────
Claude: "我們已經拆分成 5 個故事。讓我們先專注在 Story 1.1（使用者登入）。"

Claude: "對於使用者登入，在什麼情況下會成功？"
User: "帳號和密碼正確..."
Claude: "好的，讓我們深入探索這個規則..."
[完整的 Example Mapping]

Claude: "Story 1.1 的 Example Mapping 已完成，準備度為綠燈。"
Claude: "你可以開始開發 Story 1.1 了。"
Claude: "接下來要進行 Story 1.2 的 Example Mapping 嗎？"
```

**注意**：此技能專注於 BDD 流程的「推測」和「說明」階段。完成後，使用者應該：

1. 編寫 Gherkin 場景（制定階段）
2. 實現步驟定義（自動化階段）
3. 執行測試（展示階段）
4. 業務驗收（驗證階段）

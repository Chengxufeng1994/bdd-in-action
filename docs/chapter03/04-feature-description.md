# 功能描述：使用者故事的藝術

## 概述

使用者故事是 BDD 中描述功能的核心方式，它以業務價值為導向，清楚表達「為誰」、「做什麼」、「為什麼」。好的使用者故事能夠促進團隊協作，確保所有人對功能有一致的理解。

---

## 使用者故事格式

### 格式一：目標導向（推薦）

```gherkin
In order to <實現某個業務目標>
As a <利益相關者>
I want <某個功能>
```

**特點**：

- ✅ 強調業務價值（In order to）
- ✅ 價值優先，功能次之
- ✅ 幫助團隊思考「為什麼」

**範例**：

```gherkin
In order to 管理我的資金流動
As a 銀行客戶
I want 在我的帳戶間轉帳
```

### 格式二：角色導向

```gherkin
As a <利益相關者>
I want <某個功能>
So that <我可以實現某個業務目標>
```

**特點**：

- ✅ 強調使用者角色（As a）
- ✅ 角色優先，便於理解使用者
- ✅ 最常見的格式

**範例**：

```gherkin
As a 銀行客戶
I want 在我的帳戶間轉帳
So that 我可以靈活管理我的資金
```

---

## 標準使用者故事模板

```gherkin
Feature: <功能名稱>

  <功能描述（可選）>

  As a <利益相關者>
  I want <某個功能>
  So that <我可以實現某個業務目標>

  <補充說明（可選）>
```

### 三個關鍵元素

| 元素        | 說明             | 問題                                     |
| ----------- | ---------------- | ---------------------------------------- |
| **As a**    | 誰需要這個功能？ | 利益相關者是誰？他們的角色是什麼？       |
| **I want**  | 需要什麼功能？   | 具體的功能或能力是什麼？                 |
| **So that** | 為什麼需要？     | 這個功能解決什麼業務問題？創造什麼價值？ |

## 完整範例

### 範例 1：銀行轉帳

```gherkin
Feature: 銀行帳戶間轉帳

  為了讓客戶能夠方便地管理他們的資金，
  我們提供帳戶間轉帳功能。

  As a 銀行客戶
  I want 在我的不同帳戶之間轉移資金
  So that 我可以靈活調配資金，滿足不同的財務需求

  Background:
    Given 我已登入網路銀行
    And 我擁有多個銀行帳戶

  Scenario: 成功轉帳到儲蓄帳戶
    Given 我的支票帳戶有 $1,000
    And 我的儲蓄帳戶有 $2,000
    When 我從支票帳戶轉 $100 到儲蓄帳戶
    Then 我的支票帳戶應該有 $900
    And 我的儲蓄帳戶應該有 $2,100
    And 我應該收到轉帳成功的確認
```

### 範例 2：賺取利息

```gherkin
Feature: 儲蓄帳戶利息計算

  In order to 鼓勵客戶儲蓄
  As a 銀行
  I want 為不同類型的帳戶提供競爭力的利率

  As a 銀行客戶
  I want 根據我的帳戶類型自動獲得利息
  So that 我的存款可以增值

  Scenario Outline: 不同帳戶類型的利息計算
    Given Tess 有一個 <帳戶類型> 帳戶，餘額為 $10,000
    And <帳戶類型> 帳戶的年利率是 <利率>%
    When 計算月利息
    Then 她應該獲得 $<月利息> 的利息
    And 她的帳戶餘額應該變成 $<新餘額>

    Examples:
      | 帳戶類型 | 利率 | 月利息 | 新餘額    |
      | 支票帳戶 | 1.0  | 8.33   | 10,008.33 |
      | 儲蓄帳戶 | 3.0  | 25.00  | 10,025.00 |
      | 超級儲蓄 | 5.0  | 41.67  | 10,041.67 |
```

### 範例 3：線上購物

```gherkin
Feature: 購物車管理

  As a 線上購物者
  I want 將商品加入購物車並隨時修改
  So that 我可以在決定購買前整理我的訂單

  Background:
    Given 我在線上商店的首頁
    And 我已經登入我的帳戶

  Scenario: 加入商品到購物車
    Given 我正在瀏覽「筆記型電腦」分類
    When 我點擊「MacBook Pro」的「加入購物車」按鈕
    Then 我應該看到「商品已加入購物車」的訊息
    And 購物車圖示應該顯示「1」個商品

  Scenario: 修改購物車數量
    Given 我的購物車中有 1 件「MacBook Pro」
    When 我將數量更改為 2
    Then 購物車應該顯示 2 件「MacBook Pro」
    And 總價應該更新為 $5,998
```

## 撰寫使用者故事的 INVEST 原則

好的使用者故事應該符合 INVEST 原則：

### I - Independent（獨立的）

**定義**：故事之間應該盡可能獨立，可以任意順序開發

✅ **好的範例**：

```
Story 1: As a 客戶, I want 轉帳到儲蓄帳戶
Story 2: As a 客戶, I want 查看交易歷史
```

❌ **避免**：

```
Story 1: As a 客戶, I want 登入系統
Story 2: As a 客戶, I want 登入後轉帳（依賴 Story 1）
```

### N - Negotiable（可協商的）

**定義**：細節可以與團隊協商，不是固定的合約

✅ **好的範例**：

```gherkin
As a 客戶
I want 快速完成轉帳
So that 我可以節省時間
# 「快速」的定義可以討論：3秒？5秒？
```

### E - Valuable（有價值的）

**定義**：對業務或使用者有明確價值

✅ **好的範例**：

```gherkin
As a 銀行
I want 自動化利息計算
So that 減少人工錯誤並降低營運成本
# 明確的業務價值
```

❌ **避免**：

```gherkin
As a 開發者
I want 重構資料庫查詢
So that 代碼更乾淨
# 這是技術任務，不是使用者故事
```

### S - Estimable（可估算的）

**定義**：團隊能夠估算工作量

✅ **好的範例**：

```gherkin
As a 客戶
I want 轉帳到儲蓄帳戶
So that 我可以管理資金
# 範圍明確，可以估算
```

❌ **避免**：

```gherkin
As a 客戶
I want 完整的財務管理系統
So that 我可以管理所有財務
# 太大、太模糊，無法估算
```

### T - Small（小的）

**定義**：足夠小，可以在一個 Sprint 內完成

✅ **好的範例**：

```gherkin
As a 客戶
I want 查看我的當前帳戶餘額
So that 我知道有多少錢可用
# 小而具體
```

❌ **避免**：

```gherkin
As a 客戶
I want 完整的網路銀行平台
So that 我可以線上管理所有銀行業務
# 太大，需要拆分成多個故事
```

### T - Testable（可測試的）

**定義**：有明確的驗收標準

✅ **好的範例**：

```gherkin
As a 客戶
I want 轉帳限額提醒
So that 我不會超過每日限額

Acceptance Criteria:
- 當轉帳金額超過剩餘限額時顯示警告
- 顯示當日已使用的轉帳額度
- 顯示剩餘可用額度
```

## 利益相關者角色範例

### 常見角色類型

| 角色類型       | 範例                   | 說明                 |
| -------------- | ---------------------- | -------------------- |
| **終端使用者** | 銀行客戶、購物者、學生 | 直接使用系統的人     |
| **系統管理員** | 系統管理員、客服人員   | 管理和維護系統的人   |
| **業務角色**   | 銀行經理、店長、教師   | 業務流程中的關鍵角色 |
| **系統**       | 系統、排程器、通知服務 | 自動化處理的系統組件 |

### 具體範例

```gherkin
# 終端使用者
As a 註冊用戶
I want 重設我的密碼
So that 我忘記密碼時仍能訪問我的帳戶

# 系統管理員
As a 系統管理員
I want 查看所有失敗的轉帳記錄
So that 我可以識別和解決系統問題

# 業務角色
As a 分行經理
I want 查看每日轉帳報表
So that 我可以監控異常交易

# 系統
As a 系統
I want 每晚自動計算所有帳戶的利息
So that 客戶可以及時獲得利息收益
```

## 常見錯誤與改進

### 錯誤 1：過於技術化

❌ **錯誤**：

```gherkin
As a 開發者
I want RESTful API 端點
So that 前端可以調用
```

✅ **改進**：

```gherkin
As a 行動應用使用者
I want 在手機上查看我的帳戶餘額
So that 我隨時都能掌握財務狀況
```

### 錯誤 2：缺乏業務價值

❌ **錯誤**：

```gherkin
As a 使用者
I want 一個按鈕
So that 我可以點擊它
```

✅ **改進**：

```gherkin
As a 銀行客戶
I want 快速訪問常用功能
So that 我可以高效完成日常銀行業務
```

### 錯誤 3：太大無法實現

❌ **錯誤**：

```gherkin
As a 客戶
I want 完整的財務管理解決方案
So that 我可以管理所有財務
```

✅ **改進**（拆分成多個小故事）：

```gherkin
Story 1:
As a 客戶
I want 查看所有帳戶餘額總覽
So that 我可以了解整體財務狀況

Story 2:
As a 客戶
I want 設定支出預算
So that 我可以控制開支

Story 3:
As a 客戶
I want 查看月度支出分析
So that 我可以了解消費習慣
```

### 錯誤 4：包含實現細節

❌ **錯誤**：

```gherkin
As a 客戶
I want 使用 React 組件顯示帳戶資訊
So that 介面很漂亮
```

✅ **改進**：

```gherkin
As a 客戶
I want 清楚易懂的帳戶資訊展示
So that 我可以快速理解我的財務狀況
```

## 撰寫工作坊流程

### 使用者故事工作坊（60 分鐘）

#### 第一階段：腦力激盪（15 分鐘）

1. **識別利益相關者**

   ```
   誰會使用這個系統？
   誰會從這個功能中受益？
   誰會受到影響？
   ```

2. **列出需求**

   ```
   每個角色需要什麼？
   他們想要完成什麼任務？
   ```

3. **識別業務價值**
   ```
   為什麼他們需要這個？
   這解決了什麼問題？
   創造了什麼價值？
   ```

#### 第二階段：編寫故事（20 分鐘）

使用便利貼，每張寫一個使用者故事：

```
┌─────────────────────────────────┐
│ As a [角色]                      │
│ I want [功能]                    │
│ So that [價值]                   │
│                                  │
│ 優先級: High/Medium/Low          │
│ 估算: S/M/L                      │
└─────────────────────────────────┘
```

#### 第三階段：優先級排序（15 分鐘）

使用 MoSCoW 方法：

- **Must have** - 必須有
- **Should have** - 應該有
- **Could have** - 可以有
- **Won't have** - 不會有（這次）

#### 第四階段：驗收標準（10 分鐘）

為高優先級故事定義驗收標準：

```gherkin
Given [前置條件]
When [執行動作]
Then [預期結果]
```

## 驗收標準範本

### 基本格式

```gherkin
Given [初始狀態]
When [觸發事件]
Then [預期結果]
And [額外結果]
```

### 完整範例

```gherkin
Feature: 帳戶間轉帳

  As a 銀行客戶
  I want 在我的帳戶間轉移資金
  So that 我可以靈活管理我的財務

  Acceptance Criteria:

  Scenario 1: 成功轉帳
    Given 我的支票帳戶有足夠的餘額
    When 我轉帳到儲蓄帳戶
    Then 轉帳應該成功
    And 兩個帳戶的餘額應該正確更新
    And 我應該收到確認通知

  Scenario 2: 餘額不足
    Given 我的支票帳戶餘額不足
    When 我嘗試轉帳
    Then 轉帳應該被拒絕
    And 我應該看到「餘額不足」的錯誤訊息
    And 帳戶餘額不應該改變

  Scenario 3: 超過限額
    Given 我已達到每日轉帳限額
    When 我嘗試再次轉帳
    Then 轉帳應該被拒絕
    And 我應該看到「超過每日限額」的訊息
    And 系統應該顯示我的剩餘限額
```

## 最佳實踐檢查清單

### ✅ 使用者故事品質檢查

- [ ] 使用業務語言，非技術術語
- [ ] 清楚指明利益相關者（As a）
- [ ] 明確說明需要的功能（I want）
- [ ] 解釋業務價值（So that）
- [ ] 符合 INVEST 原則
- [ ] 有明確的驗收標準
- [ ] 可以在一個 Sprint 內完成
- [ ] 所有人都能理解
- [ ] 可以獨立測試
- [ ] 有明確的優先級

### ✅ 驗收標準檢查

- [ ] 使用 Given-When-Then 格式
- [ ] 涵蓋正常流程
- [ ] 包含錯誤情況
- [ ] 包含邊界案例
- [ ] 可測試和可驗證
- [ ] 具體而非抽象
- [ ] 完整且無歧義

## 總結

### 關鍵要點

1. **以價值為中心**
   - 始終關注「為什麼」
   - 業務價值驅動功能開發

2. **使用者導向**
   - 從使用者角度描述
   - 使用業務語言

3. **可測試性**
   - 明確的驗收標準
   - Given-When-Then 場景

4. **適當大小**
   - 符合 INVEST 原則
   - 可在一個迭代內完成

5. **協作產出**
   - 三方協作編寫
   - 持續精煉和更新

### 下一步

1. 選擇一個功能練習編寫使用者故事
2. 組織使用者故事工作坊
3. 為故事定義驗收標準
4. 轉化為 Gherkin 場景
5. 開始實現和測試

---

**記住：好的使用者故事是團隊協作的起點，不是終點。持續對話和精煉才能確保正確理解需求。** 💡

# 故事切分：將功能分解為可管理的使用者故事

## 概述

故事切分（Story Splitting）是將大型功能或 Epic 分解為小型、可交付的使用者故事的技術。好的故事切分能夠：

- 🎯 **加速交付**：小故事更快完成
- 🔄 **提早反饋**：盡早展示價值
- 📊 **降低風險**：減少不確定性
- 🤝 **促進協作**：團隊更容易理解和估算

---

## 為什麼要切分故事？

### 問題：故事太大

```
❌ Epic：完整的網路銀行系統

工作量：無法估算
時間：數月
風險：極高
價值：延遲實現
```

### 解決：切分成小故事

```
✅ Story 1：查看帳戶餘額
✅ Story 2：查看交易歷史
✅ Story 3：帳戶間轉帳
✅ Story 4：設定轉帳限額
✅ Story 5：轉帳通知

工作量：可估算（每個 2-5 天）
時間：每週交付價值
風險：可控
價值：持續實現
```

---

## 何時需要切分？

### 故事太大的徵兆

| 徵兆 | 說明 | 建議 |
|------|------|------|
| **無法估算** | 團隊無法給出確定的工作量估算 | 立即切分 |
| **超過 Sprint** | 無法在一個迭代內完成 | 必須切分 |
| **太多規則** | Example Mapping 超過 8 個規則 | 建議切分 |
| **太多問題** | Example Mapping 紅色便利貼 > 6 個 | 需要切分 |
| **超時討論** | Example Mapping 超過 30 分鐘 | 故事太大 |
| **難以測試** | 無法定義清晰的驗收標準 | 考慮切分 |

### INVEST 原則檢查

故事應該是：
- **S** (Small) - 小的：可在 1-2 天內完成
- **I** (Independent) - 獨立的：不依賴其他故事

如果不符合，就需要切分！

## 故事切分模式

### 模式 1：工作流程步驟（Workflow Steps）

**適用**：有明確流程的功能

**範例**：轉帳功能

```
❌ 原始故事（太大）：
As a 客戶
I want 完整的轉帳功能
So that 我可以管理資金

✅ 切分後：

Story 1：選擇帳戶
As a 客戶
I want 選擇來源和目標帳戶
So that 我可以開始轉帳流程

Story 2：輸入金額
As a 客戶
I want 輸入轉帳金額
So that 我可以指定要轉多少錢

Story 3：確認轉帳
As a 客戶
I want 查看並確認轉帳詳情
So that 我可以確保資訊正確

Story 4：執行轉帳
As a 客戶
I want 執行轉帳並接收確認
So that 我知道轉帳成功
```

**流程圖**：
```
選擇帳戶 → 輸入金額 → 確認詳情 → 執行轉帳
  Story 1    Story 2     Story 3     Story 4
```

---

### 模式 2：業務規則變化（Business Rule Variations）

**適用**：不同條件下行為不同

**範例**：帳戶類型

```
❌ 原始故事（太大）：
As a 客戶
I want 開立任何類型的帳戶
So that 我可以管理不同的資金

✅ 切分後：

Story 1：開立支票帳戶（最簡單，最常用）
As a 客戶
I want 開立支票帳戶
So that 我可以日常支付

Story 2：開立儲蓄帳戶
As a 客戶
I want 開立儲蓄帳戶
So that 我可以儲蓄並賺取利息

Story 3：開立投資帳戶（複雜度最高）
As a 客戶
I want 開立投資帳戶
So that 我可以投資理財
```

**優先級**：支票（高） > 儲蓄（中） > 投資（低）

---

### 模式 3：主要工作（Major Effort）

**適用**：技術複雜度高的功能

**範例**：報表生成

```
❌ 原始故事（太大）：
As a 經理
I want 完整的財務報表系統
So that 我可以分析業務

✅ 切分後：

Story 1：簡單報表（骨架）
As a 經理
I want 查看基本的帳戶摘要
So that 我可以快速了解概況

Story 2：詳細報表（肉）
As a 經理
I want 查看詳細的交易記錄
So that 我可以深入分析

Story 3：匯出功能（皮）
As a 經理
I want 匯出報表為 PDF
So that 我可以分享給他人

Story 4：排程報表（附加價值）
As a 經理
I want 設定自動生成報表
So that 我可以定期接收
```

**Walking Skeleton**：先實現骨架（基本功能），再逐步添加

---

### 模式 4：簡單/複雜（Simple/Complex）

**適用**：有明顯難度差異

**範例**：搜尋功能

```
❌ 原始故事（太大）：
As a 客戶
I want 強大的搜尋功能
So that 我可以找到任何交易

✅ 切分後：

Story 1：簡單搜尋（基本）
As a 客戶
I want 按日期搜尋交易
So that 我可以找到特定期間的交易

Story 2：多條件搜尋（中等）
As a 客戶
I want 按日期和金額範圍搜尋
So that 我可以更精確地過濾

Story 3：全文搜尋（複雜）
As a 客戶
I want 按描述或收款人搜尋
So that 我可以找到任何相關交易

Story 4：進階過濾（最複雜）
As a 客戶
I want 儲存和重用搜尋條件
So that 我可以快速執行常用搜尋
```

---

### 模式 5：資料變化（Variations in Data）

**適用**：處理不同類型的資料

**範例**：通知系統

```
❌ 原始故事（太大）：
As a 客戶
I want 接收所有類型的通知
So that 我可以掌握帳戶動態

✅ 切分後：

Story 1：Email 通知（最基本）
As a 客戶
I want 透過 Email 接收轉帳通知
So that 我知道資金變動

Story 2：SMS 通知
As a 客戶
I want 透過 SMS 接收重要通知
So that 我可以即時知道

Story 3：推送通知
As a 客戶
I want 在手機 App 接收推送
So that 我可以即時查看

Story 4：通知偏好設定
As a 客戶
I want 自訂通知方式和頻率
So that 我可以控制接收方式
```

### 模式 6：操作（CRUD Operations）

**適用**：標準的 CRUD 功能

**範例**：收款人管理

```
❌ 原始故事（太大）：
As a 客戶
I want 管理我的收款人清單
So that 我可以快速轉帳

✅ 切分後：

Story 1：新增收款人（Create）
As a 客戶
I want 新增收款人到清單
So that 我可以儲存常用帳戶

Story 2：查看收款人（Read）
As a 客戶
I want 查看我的收款人清單
So that 我可以選擇轉帳對象

Story 3：修改收款人（Update）
As a 客戶
I want 修改收款人資訊
So that 我可以更新變更的資訊

Story 4：刪除收款人（Delete）
As a 客戶
I want 刪除不再使用的收款人
So that 我可以保持清單整潔
```

**優先級**：C > R > U > D（通常這個順序）

---

### 模式 7：介面（Interface）

**適用**：多平台或裝置

**範例**：多平台支援

```
❌ 原始故事（太大）：
As a 客戶
I want 在任何裝置上使用銀行服務
So that 我隨時隨地都能操作

✅ 切分後：

Story 1：網頁版（桌面）
As a 客戶
I want 在電腦瀏覽器上操作
So that 我可以在辦公室使用

Story 2：響應式設計（手機瀏覽器）
As a 客戶
I want 在手機瀏覽器上使用
So that 我不需要安裝 App

Story 3：原生 App（iOS）
As a 客戶
I want 使用 iOS App
So that 我可以獲得最佳體驗

Story 4：原生 App（Android）
As a 客戶
I want 使用 Android App
So that 我的裝置也能使用
```

---

### 模式 8：效能/規模（Performance/Scale）

**適用**：效能要求不同

**範例**：批次處理

```
❌ 原始故事（太大）：
As a 系統
I want 處理任意數量的批次交易
So that 可以支援大量操作

✅ 切分後：

Story 1：小批次（< 100 筆）
As a 系統
I want 處理少量批次交易
So that 可以支援基本需求

Story 2：中批次（100-1000 筆）
As a 系統
I want 處理中等批次交易
So that 可以支援日常作業

Story 3：大批次（> 1000 筆）
As a 系統
I want 處理大量批次交易
So that 可以支援企業客戶

Story 4：效能優化
As a 系統
I want 優化批次處理效能
So that 可以更快完成
```

## 故事切分技巧

### 技巧 1：垂直切分 vs 水平切分

#### ❌ 水平切分（避免）

按技術層次切分，每個故事無法獨立交付價值：

```
Story 1：設計資料庫 schema
Story 2：建立 API 端點
Story 3：實現前端介面
Story 4：撰寫測試
```

**問題**：
- 每個故事無法獨立展示
- 無法提早獲得反饋
- 風險集中在最後

#### ✅ 垂直切分（推薦）

每個故事都包含完整的功能切片：

```
Story 1：查看帳戶餘額（含 DB + API + UI + Test）
Story 2：查看交易歷史（含 DB + API + UI + Test）
Story 3：執行轉帳（含 DB + API + UI + Test）
```

**優點**：
- 每個故事可獨立展示
- 盡早獲得使用者反饋
- 風險分散
- 持續交付價值

### 技巧 2：使用故事對映（Story Mapping）

```
使用者旅程
════════════════════════════════════════════════════════

登入 → 查看帳戶 → 選擇操作 → 轉帳 → 確認 → 登出
 │       │          │         │      │      │
 ↓       ↓          ↓         ↓      ↓      ↓

MVP (必須有)
─────────────────────────────────────────────────────
S1     S2                    S3
登入   查看餘額              簡單轉帳

Version 2 (應該有)
─────────────────────────────────────────────────────
              S4         S5         S6
              交易歷史   選擇收款人 Email 通知

Version 3 (可以有)
─────────────────────────────────────────────────────
              S7         S8         S9      S10
              報表       批次轉帳   排程    推送通知
```

### 技巧 3：Spike 故事

對於不確定性高的功能，先做探索：

```
Spike Story：
研究 OAuth 整合的可行性
時間盒：2 天
產出：技術方案建議

接著：
Story 1：實現基本 OAuth 登入
Story 2：處理 Token 刷新
Story 3：整合使用者資訊
```

## 切分工作坊流程

### 步驟 1：識別大故事（10 分鐘）

```
團隊：這個故事太大了嗎？

檢查：
□ 估算 > 13 點
□ 無法在一個 Sprint 完成
□ Example Mapping 超時
□ 規則 > 8 個
□ 問題 > 6 個

如果任一項為 Yes → 需要切分
```

### 步驟 2：選擇切分模式（5 分鐘）

```
討論：哪種切分模式最適合？

□ 工作流程步驟
□ 業務規則變化
□ 簡單/複雜
□ 資料變化
□ CRUD 操作
□ 介面
□ 效能/規模

可能使用多種模式組合
```

### 步驟 3：應用模式切分（20 分鐘）

```
使用便利貼：

1. 寫下所有可能的小故事
2. 每張便利貼一個故事
3. 確保每個都是完整的垂直切片
4. 檢查 INVEST 原則
```

### 步驟 4：排序優先級（10 分鐘）

```
使用 MoSCoW 方法：

Must have (必須有) - Sprint 1
Should have (應該有) - Sprint 2
Could have (可以有) - Sprint 3
Won't have (暫不做) - Backlog
```

### 步驟 5：驗證切分（5 分鐘）

```
檢查每個故事：

□ 獨立的（不依賴其他故事）
□ 可估算的（2-5 天）
□ 有價值的（可獨立展示）
□ 可測試的（清晰驗收標準）

如果都符合 → 切分成功！
```

## 切分範例：完整示範

### 原始 Epic

```
Epic：網路銀行轉帳系統

As a 銀行客戶
I want 完整的線上轉帳系統
So that 我可以管理所有的資金轉移需求

估算：無法估算（太大）
問題：範圍不明確
```

### 第一次切分：按工作流程

```
Story 1：選擇帳戶
Story 2：輸入轉帳資訊
Story 3：驗證和確認
Story 4：執行轉帳
Story 5：查看結果

評估：Story 2 和 3 仍然太大
```

```
Story 1.1：查看帳戶餘額
As a 客戶
I want 查看我的帳戶餘額
So that 我知道有多少錢可用
估算：3 點
優先級：Must have

Story 1.2：簡單轉帳（同幣別，小金額）
As a 客戶
I want 在我的帳戶間轉 ≤ $1,000
So that 我可以調配資金
估算：5 點
優先級：Must have
限制：只支援同幣別、單筆限額 $1,000

Version 2 (Sprint 2)
─────────────────────────────────────────────

Story 2.1：提高轉帳限額
As a 客戶
I want 轉帳最多 $10,000
So that 我可以轉移更多資金
估算：2 點
優先級：Should have

Story 2.2：轉帳通知
As a 客戶
I want 接收轉帳成功的 Email
So that 我可以確認交易
估算：3 點
優先級：Should have

Story 2.3：轉帳歷史
As a 客戶
I want 查看轉帳記錄
So that 我可以追蹤過去的交易
估算：5 點
優先級：Should have

Version 3 (Sprint 3)
─────────────────────────────────────────────

Story 3.1：收款人清單
As a 客戶
I want 儲存常用收款人
So that 我可以快速選擇
估算：3 點
優先級：Could have

Story 3.2：排程轉帳
As a 客戶
I want 設定未來日期的轉帳
So that 我可以自動化定期付款
估算：8 點
優先級：Could have

Story 3.3：跨幣別轉帳
As a 客戶
I want 在不同幣別的帳戶間轉帳
So that 我可以換匯

```

### 切分結果評估

```
✅ 成功指標：

• 總故事數：7
• MVP 故事數：2（可在 1 個 Sprint 完成）
• 每個故事：2-8 點（可估算）
• 垂直切分：每個都可獨立展示
• 優先級明確：Must > Should > Could > Won't
• 風險降低：從簡單到複雜漸進

📊 交付時間線：

Sprint 1：基本查看 + 簡單轉帳
Sprint 2：提高限額 + 通知 + 歷史
Sprint 3：收款人 + 排程
未來：跨幣別
```

## 常見錯誤與解決

### 錯誤 1：切分過細

❌ **問題**：
```
Story 1：建立轉帳按鈕
Story 2：實現按鈕點擊
Story 3：打開轉帳對話框
Story 4：輸入金額欄位
Story 5：顯示確認訊息
```

✅ **解決**：
```
Story：基本轉帳功能
包含：按鈕、對話框、輸入、確認
完整的垂直功能切片
```

### 錯誤 2：水平切分

❌ **問題**：
```
Story 1：資料庫設計
Story 2：API 開發
Story 3：前端介面
```

✅ **解決**：
```
Story：查看帳戶餘額
包含：DB schema + API + UI
每個故事都是完整功能
```

### 錯誤 3：依賴太多

❌ **問題**：
```
Story 2 依賴 Story 1
Story 3 依賴 Story 2
Story 4 依賴 Story 1, 2, 3
```

✅ **解決**：
```
重新切分，讓每個故事盡可能獨立
或調整順序，讓依賴最小化
```

### 錯誤 4：缺乏業務價值

❌ **問題**：
```
Story：重構資料庫查詢
（技術改進，無業務價值）
```

✅ **解決**：
```
Story：提升帳戶查詢速度
As a 客戶
I want 快速查看帳戶資訊
So that 我可以節省時間

技術任務：作為實現的一部分
```

## 切分檢查清單

### ✅ 切分前

- [ ] 確認故事太大（估算 > 13 點或無法估算）
- [ ] 識別適用的切分模式
- [ ] 召集 Three Amigos（業務、開發、測試）
- [ ] 準備故事對映或白板

### ✅ 切分中

- [ ] 應用至少一種切分模式
- [ ] 確保垂直切分（不是水平）
- [ ] 每個小故事都有業務價值
- [ ] 檢查 INVEST 原則
- [ ] 排定優先級

### ✅ 切分後

- [ ] 每個故事可在 1-2 天完成
- [ ] 每個故事可獨立測試
- [ ] 每個故事可獨立展示
- [ ] 依賴關係最小化
- [ ] 有明確的驗收標準

## 總結

### 關鍵要點

1. **垂直切分優先**
   - 每個故事都是完整的功能切片
   - 包含所有層次（DB + API + UI）

2. **保持價值**
   - 每個故事都要有業務價值
   - 可以獨立展示給利益相關者

3. **漸進交付**
   - 從 MVP 開始
   - 逐步增加複雜度

4. **符合 INVEST**
   - Independent（獨立的）
   - Negotiable（可協商的）
   - Valuable（有價值的）
   - Estimable（可估算的）
   - Small（小的）
   - Testable（可測試的）

### 切分模式速查

| 模式 | 適用場景 | 範例 |
|------|----------|------|
| 工作流程步驟 | 有明確流程 | 登入 → 選擇 → 執行 → 確認 |
| 業務規則變化 | 不同條件 | 不同帳戶類型 |
| 簡單/複雜 | 難度差異 | 基本搜尋 → 進階搜尋 |
| 資料變化 | 不同資料 | Email → SMS → 推送 |
| CRUD 操作 | 標準功能 | 新增 → 讀取 → 更新 → 刪除 |
| 介面 | 多平台 | Web → Mobile → App |
| 效能/規模 | 效能要求 | 小批次 → 大批次 |

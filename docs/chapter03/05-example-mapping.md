# Example Mapping：發現需求的協作技術

## 概述

Example Mapping 是一種輕量級的協作技術，用於探索和理解使用者故事。透過使用四種顏色的便利貼，團隊可以快速識別規則、範例和問題，確保對需求有共同的理解。

---

## 什麼是 Example Mapping？

Example Mapping 是由 Matt Wynne 於 2015 年提出的一種結構化對話技術。它幫助團隊：

- 🎯 **快速探索**需求的複雜度
- 💡 **發現盲點**和未知的問題
- 🤝 **建立共識**確保理解一致
- ⏱️ **節省時間**避免過度規劃

### 核心原則

1. **時間盒限制**：25-30 分鐘
2. **三方協作**：業務、開發、測試（Three Amigos）
3. **具體範例**：用實際案例說明規則
4. **視覺化組織**：使用顏色編碼的便利貼

---

## 四色便利貼系統

```
📋 黃色 - 使用者故事
    ↓
📏 藍色 - 業務規則
    ↓
📝 綠色 - 具體範例
    ↓
❓ 紅色 - 待解決問題
```

### 1. 📋 黃色：使用者故事

**用途**：描述功能的整體目標

**內容**：
```
As a <角色>
I want <功能>
So that <價值>
```

**範例**：
```
📋 As a 銀行客戶
   I want 在帳戶間轉帳
   So that 我可以管理資金
```

### 2. 📏 藍色：業務規則

**用途**：定義功能的規則和約束

**提示詞**：
- "在什麼情況下..."
- "如果...那麼..."
- "只有當..."
- "除非..."

**範例**：
```
📏 規則 1：基本轉帳驗證
   - 來源帳戶必須有足夠餘額
   - 目標帳戶必須是有效的

📏 規則 2：轉帳限額
   - 單筆轉帳不得超過 $10,000
   - 每日轉帳總額不得超過 $50,000

📏 規則 3：帳戶類型限制
   - 投資帳戶無法轉出
   - 儲蓄帳戶每月轉出限 6 次
```

### 3. 📝 綠色：具體範例

**用途**：用實際數據說明規則

**格式**：Given-When-Then

**範例**：
```
📝 範例 1：成功轉帳
   Given 支票帳戶有 $1,000
   When 轉 $100 到儲蓄帳戶
   Then 支票帳戶剩 $900
   And 儲蓄帳戶變成 $2,100

📝 範例 2：餘額不足
   Given 支票帳戶有 $50
   When 嘗試轉 $100
   Then 顯示「餘額不足」錯誤
   And 帳戶餘額不變
```

### 4. ❓ 紅色：待解決問題

**用途**：記錄不確定或需要澄清的事項

**範例**：
```
❓ 轉帳是即時的還是有延遲？
❓ 跨幣別轉帳的匯率如何計算？
❓ 是否需要雙重認證？
❓ 轉帳失敗後是否需要通知？
```

## Example Mapping 工作坊流程

### 準備工作（5 分鐘前）

#### 材料清單
- [ ] 四色便利貼（黃、藍、綠、紅）
- [ ] 白板或大桌面
- [ ] 馬克筆
- [ ] 計時器

#### 參與者
- **業務分析師** - 主持人，負責業務規則
- **開發人員** - 提出技術問題和可行性
- **測試人員** - 關注邊界案例和測試場景
- **產品負責人**（可選）- 最終決策者

#### 環境設置
```
┌──────────────────────────────────────┐
│         白板或桌面                    │
│                                      │
│  📋 [使用者故事]                     │
│                                      │
│  📏 [規則1]  📏 [規則2]  📏 [規則3]  │
│   ├─ 📝 範例    ├─ 📝 範例           │
│   ├─ 📝 範例    ├─ 📝 範例           │
│   └─ ❓ 問題    └─ ❓ 問題           │
│                                      │
│  ❓ [一般性問題]                     │
└──────────────────────────────────────┘
```

---

### 第一階段：設定背景（5 分鐘）

#### 1. 展示使用者故事

主持人在白板頂端貼上黃色便利貼：

```
📋 As a 銀行客戶
   I want 在帳戶間轉帳
   So that 我可以管理資金
```

#### 2. 朗讀故事

確保所有人理解故事的：
- **誰**：利益相關者是誰？
- **什麼**：想要什麼功能？
- **為什麼**：創造什麼價值？

#### 3. 設定時間盒

```
⏱️ 開始時間：14:00
⏱️ 結束時間：14:25
⏱️ 持續時間：25 分鐘
```

**重要**：如果超時，故事可能太大，需要拆分！

---

### 第二階段：探索規則（10 分鐘）

#### 引導問題

1. **識別規則**
   ```
   Q: "在什麼情況下轉帳會成功？"
   A: 📏 規則：餘額必須足夠

   Q: "有哪些限制或約束？"
   A: 📏 規則：單筆不超過 $10,000

   Q: "什麼時候會失敗？"
   A: 📏 規則：餘額不足時拒絕
   ```

2. **驗證規則**
   ```
   Q: "這個規則總是適用嗎？"
   Q: "有例外情況嗎？"
   Q: "不同帳戶類型規則一樣嗎？"
   ```

3. **組織規則**

   將相關規則分組：
   ```
   驗證類規則        限額類規則        通知類規則
   📏 餘額檢查       📏 單筆限額       📏 成功通知
   📏 帳戶驗證       📏 每日限額       📏 失敗通知
   📏 狀態檢查       📏 月度限額       📏 異常警示
   ```

#### 實例：轉帳功能

```
📋 使用者故事：帳戶間轉帳

📏 規則 1：基本驗證
   - 來源帳戶餘額必須 ≥ 轉帳金額
   - 來源和目標帳戶必須都是有效的
   - 來源和目標帳戶必須不同

📏 規則 2：轉帳限額
   - 單筆轉帳：≤ $10,000
   - 每日總額：≤ $50,000
   - VIP 客戶限額更高

📏 規則 3：帳戶類型
   - 支票帳戶：可自由轉入轉出
   - 儲蓄帳戶：每月轉出限 6 次
   - 投資帳戶：只能轉入，不能轉出

📏 規則 4：手續費
   - 同類型帳戶：免費
   - 不同類型：收取 $2 手續費
   - VIP 客戶：全部免費
```

---

### 第三階段：創建範例（10 分鐘）

#### 為每個規則創建範例

**原則**：
- ✅ 至少 2 個範例（正常 + 異常）
- ✅ 使用具體數據
- ✅ 涵蓋邊界情況
- ✅ Given-When-Then 格式

#### 規則 1 的範例

```
📏 規則 1：基本驗證

📝 範例 1.1：成功轉帳（正常情況）
   Given 支票帳戶有 $1,000
   And 儲蓄帳戶有 $500
   When 從支票轉 $200 到儲蓄
   Then 支票帳戶剩 $800
   And 儲蓄帳戶變成 $700

📝 範例 1.2：餘額不足（異常情況）
   Given 支票帳戶有 $100
   When 嘗試轉 $200 到儲蓄
   Then 顯示「餘額不足」錯誤
   And 帳戶餘額不變

📝 範例 1.3：相同帳戶（邊界情況）
   Given 支票帳戶 A
   When 嘗試從 A 轉到 A
   Then 顯示「不能轉給自己」錯誤
```

#### 規則 2 的範例

```
📏 規則 2：轉帳限額

📝 範例 2.1：在限額內
   Given 支票帳戶有 $20,000
   When 轉 $9,999 到儲蓄
   Then 轉帳成功

📝 範例 2.2：超過單筆限額
   Given 支票帳戶有 $20,000
   When 嘗試轉 $10,001
   Then 顯示「超過單筆限額 $10,000」錯誤

📝 範例 2.3：超過每日限額
   Given 今日已轉帳 $45,000
   And 每日限額是 $50,000
   When 嘗試轉 $6,000
   Then 顯示「超過每日限額」錯誤
   And 顯示「今日剩餘額度 $5,000」

📝 範例 2.4：VIP 客戶
   Given VIP 客戶帳戶有 $100,000
   When 轉 $15,000（超過一般限額）
   Then 轉帳成功（VIP 限額更高）
```

#### 邊界值範例

關注「剛好」的情況：

```
📝 範例：邊界值測試

• 剛好等於限額
  When 轉 $10,000（等於限額）
  Then 轉帳成功

• 剛好超過限額
  When 轉 $10,000.01
  Then 轉帳失敗

• 餘額剛好夠
  Given 帳戶有 $100
  When 轉 $100
  Then 成功，餘額變 $0

• 小數金額
  When 轉 $99.99
  Then 正確處理小數
```

---

### 第四階段：識別問題（5 分鐘）

#### 問題類型

1. **業務規則不明確**
   ```
   ❓ VIP 客戶的限額具體是多少？
   ❓ 儲蓄帳戶的「月」是自然月還是 30 天？
   ❓ 手續費從哪個帳戶扣除？
   ```

2. **技術可行性**
   ```
   ❓ 轉帳是同步還是異步處理？
   ❓ 失敗後是否自動重試？
   ❓ 如何處理併發轉帳？
   ```

3. **整合問題**
   ```
   ❓ 需要與哪些外部系統整合？
   ❓ 如何驗證目標帳戶存在？
   ❓ 交易記錄存儲在哪裡？
   ```

4. **使用者體驗**
   ```
   ❓ 轉帳成功後如何通知客戶？
   ❓ 錯誤訊息應該多詳細？
   ❓ 是否需要確認步驟？
   ```

#### 問題管理

**記錄問題**：
```
❓ 問題描述
   • 誰負責解答：[負責人]
   • 預計時間：[日期]
   • 優先級：高/中/低
   • 阻塞開發：是/否
```

**決策矩陣**：

| 問題 | 優先級 | 阻塞？ | 負責人 | 期限 |
|------|--------|--------|--------|------|
| VIP 限額 | 高 | 是 | PO | 今天 |
| 通知方式 | 中 | 否 | BA | 本週 |
| 重試機制 | 低 | 否 | Tech Lead | 下週 |

---

### 第五階段：總結決策（5 分鐘）

#### 評估標準

✅ **準備好開發**（綠燈）：
- [ ] 所有規則都有至少 2 個範例
- [ ] 涵蓋正常和異常情況
- [ ] 紅色便利貼 ≤ 3 張
- [ ] 所有人理解一致
- [ ] 可以在 1-2 個 Sprint 完成

⚠️ **需要更多工作**（黃燈）：
- [ ] 有 4-6 張紅色便利貼
- [ ] 部分規則缺少範例
- [ ] 仍有爭議或不確定性
- [ ] 需要更多研究

🛑 **故事太大**（紅燈）：
- [ ] 超過時間盒（>30 分鐘）
- [ ] 紅色便利貼 > 6 張
- [ ] 規則複雜且相互依賴
- [ ] 無法估算工作量

#### 決策行動

**綠燈**：
```
✅ 開始編寫 Gherkin 場景
✅ 分配給開發團隊
✅ 納入下個 Sprint
```

**黃燈**：
```
⚠️ 安排專門會議討論問題
⚠️ 研究技術可行性
⚠️ 與利益相關者確認規則
```

**紅燈**：
```
🛑 拆分成多個小故事
🛑 重新定義範圍
🛑 深入研究需求
```

## Example Mapping 最佳實踐

### ✅ Do（應該做的）

1. **嚴格時間盒**
   - 設定 25 分鐘計時器
   - 超時就停止
   - 超時 = 故事太大

2. **專注範例**
   - 用具體數據，不要抽象描述
   - 每個規則至少 2 個範例
   - 涵蓋正常和異常

3. **記錄問題**
   - 不要現場爭論
   - 記下問題，稍後解決
   - 指定負責人和期限

4. **視覺化組織**
   - 規則橫向排列
   - 範例垂直排在規則下
   - 相關規則靠近

5. **全員參與**
   - 鼓勵所有人發言
   - 沒有「愚蠢」的問題
   - 尊重不同觀點

### ❌ Don't（避免做的）

1. **不要過度規劃**
   - 不是寫詳細規格
   - 不是設計解決方案
   - 只是探索理解

2. **不要爭論細節**
   - 記錄為問題
   - 避免長時間討論
   - 保持節奏

3. **不要跳過範例**
   - 規則必須有範例支持
   - 抽象規則容易誤解
   - 範例是「活文檔」

4. **不要忽略紅色便利貼**
   - 問題不會自己消失
   - 未解決的問題會拖延開發
   - 必須有人負責

5. **不要獨自進行**
   - 需要不同視角
   - Three Amigos 是最低要求
   - 協作產生更好的理解

## 進階技巧

### 技巧 1：規則樹狀結構

對於複雜規則，使用層次結構：

```
📏 主規則：轉帳驗證
   ├─ 📏 子規則 1：帳戶驗證
   │   ├─ 📝 範例：有效帳戶
   │   └─ 📝 範例：無效帳戶
   │
   ├─ 📏 子規則 2：餘額驗證
   │   ├─ 📝 範例：足夠餘額
   │   └─ 📝 範例：餘額不足
   │
   └─ 📏 子規則 3：限額驗證
       ├─ 📝 範例：在限額內
       └─ 📝 範例：超過限額
```

### 技巧 2：範例模板

使用一致的範例格式：

```
📝 範例：[簡短描述]
   Given [初始狀態]
     And [更多條件]
   When [執行動作]
   Then [預期結果]
     And [額外結果]

   預期值：
   • 來源帳戶：$XXX
   • 目標帳戶：$YYY
```

### 技巧 3：顏色延伸

增加更多顏色表示優先級：


## 從 Example Mapping 到 Gherkin

### 轉換流程

```
Example Mapping          →          Gherkin Feature
══════════════                      ════════════════

📋 使用者故事            →          Feature: 描述
📏 業務規則              →          Scenario: 名稱
📝 具體範例              →          Given-When-Then
❓ 待解決問題            →          # TODO 註解或新 Story
```

### 實際轉換

**Example Mapping 輸出**：
```
📏 規則：餘額驗證

📝 範例 1：足夠餘額
   Given 支票 $1,000
   When 轉 $200
   Then 成功

📝 範例 2：餘額不足
   Given 支票 $100
   When 轉 $200
   Then 失敗
```

**Gherkin 場景**：
```gherkin
Feature: 銀行帳戶間轉帳

  Scenario: 餘額足夠時成功轉帳
    Given Tess 的支票帳戶有 $1,000

    Then 轉帳應該成功
    And 她的支票帳戶應該有 $800

  Scenario: 餘額不足時拒絕轉帳
    Given Tess 的支票帳戶有 $100
    When 她嘗試轉 $200 到儲蓄帳戶
    Then 轉帳應該失敗
    And 她應該看到「餘額不足」錯誤
```

## 常見問題 FAQ

### Q1：Example Mapping 和傳統需求文檔有什麼不同？

**A**：
- **輕量級**：25 分鐘 vs 數天編寫
- **協作式**：團隊討論 vs 一人撰寫
- **範例驅動**：具體案例 vs 抽象描述
- **及時性**：剛好夠用 vs 過度規劃

### Q2：如果 25 分鐘內沒完成怎麼辦？

**A**：故事太大，採取以下行動：
1. 立即停止會議
2. 識別核心價值
3. 拆分成多個小故事
4. 為每個小故事安排新的 Example Mapping

### Q3：紅色便利貼太多怎麼辦？

**A**：
- **< 3 個**：可以開始開發，邊做邊解決
- **3-6 個**：需要先解決再開發
- **> 6 個**：需求不清楚，需要更多研究

### Q4：沒有業務人員參加怎麼辦？

**A**：
- **不要進行**：缺少關鍵知識
- **改為研究會議**：列出需要問的問題
- **事後驗證**：與業務確認理解

### Q5：遠端團隊如何進行 Example Mapping？

**A**：
- 使用線上白板（Miro, Mural）
- 視訊會議（全程開啟攝影機）
- 準備數位便利貼模板
- 指定一人主持，控制節奏

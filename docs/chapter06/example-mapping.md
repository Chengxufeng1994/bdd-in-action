# Example Mapping：從使用者故事到可執行規格的協作技術

## 概述

Example Mapping 是 BDD Discovery 階段的核心工具，由 Matt Wynne 提出。它透過結構化的對話和視覺化組織，幫助團隊在開發前對齊理解、識別業務規則、並揭露未知（刻意探索）。

---

## 核心理念

### 刻意探索（Deliberate Discovery）

> 「在任何軟體專案中，最大的風險之一來自於自身的無知。」

Example Mapping 的存在目的，是讓團隊**主動**在開發前暴露不確定性，而不是等到開發中或上線後才發現問題。發現問題越早，修復成本越低。

### 實際選擇（Real Options）

> 「在獲得足夠資訊前，不應過早承諾特定解決方案。」

透過 Example Mapping 識別哪些問題尚未有足夠資訊（紅色便利貼），避免在資訊不足時做出錯誤決策。

---

## 四色便利貼系統

```
📋 黃色 - 使用者故事（Story）
    ↓ 由故事衍生出
📏 藍色 - 業務規則（Rule）
    ↓ 每條規則需有
📝 綠色 - 具體範例（Example）
    ↓ 討論中產生的疑問
❓ 紅色 - 待解決問題（Question）
```

| 顏色 | 代表 | 提示語 |
|------|------|--------|
| 🟡 黃色 | 使用者故事 | As a / I want / So that |
| 📘 藍色 | 業務規則 | 「在什麼情況下...」「只有當...」「如果...那麼...」 |
| 🟢 綠色 | 具體範例 | 表格格式（初始狀態 / 動作 / 預期結果） |
| 🔴 紅色 | 待解決問題 | 「我們不確定...」「需要確認...」 |

## 範例選取原則

每條業務規則至少需要三種範例：

| 類型 | 說明 | 目的 |
|------|------|------|
| **正常情況（Happy Path）** | 規則成立，動作成功 | 確認主要流程正確 |
| **異常情況（Unhappy Path）** | 規則不成立，動作失敗 | 確認錯誤處理正確 |
| **邊界情況（Edge Case）** | 恰好在臨界點 | 確認邊界行為明確 |

**範例（餘額驗證規則）：**

| 範例 | 初始狀態 | 執行動作 | 預期結果 |
|------|---------|---------|----------|
| 1.1 - 餘額充足（正常） | 支票 $1,000 | 轉 $200 | 成功，支票 $800 |
| 1.2 - 餘額不足（異常） | 支票 $100  | 轉 $200 | 失敗「餘額不足」 |
| 1.3 - 恰好等於（邊界） | 支票 $200  | 轉 $200 | 成功，支票 $0 |

## 工作坊流程（25 分鐘時間盒）

### 參與者：Three Amigos

| 角色 | 關注點 |
|------|--------|
| **業務（Business）** | 業務規則是否完整、有無遺漏 |
| **開發（Dev）** | 技術可行性、邊界條件 |
| **測試（Test）** | 異常情況、邊界值、測試覆蓋 |

### 流程步驟

```
準備（5 分鐘前）
└─> 選定一個使用者故事（黃色便利貼）

第一階段：探索業務規則（10 分鐘）
└─> 問：「在什麼情況下...？」
    「有哪些限制或約束？」
    「什麼時候會失敗？」
└─> 每條規則 → 一張藍色便利貼

第二階段：創建具體範例（10 分鐘）
└─> 對每條規則，用表格描述範例
└─> 涵蓋正常、異常、邊界情況
└─> 每個範例 → 一張綠色便利貼

第三階段：識別問題（5 分鐘）
└─> 記錄不確定的地方（紅色便利貼）
└─> 指定負責人與解決期限

評估準備度
└─> 🟢 綠燈：可進入 Formulation
└─> 🟡 黃燈：進入 Clarify 流程
└─> 🔴 紅燈：故事太大，需要拆分
```

---

## 準備度評估

| 狀態 | 條件 | 下一步 |
|------|------|--------|
| 🟢 **綠燈** | 問題數量為 0，規則和範例清楚 | 進入 Formulation（編寫 Gherkin） |
| 🟡 **黃燈** | 有少數問題待解決，主要邏輯清楚 | 進入 Clarify 循環，逐一釐清問題 |
| 🔴 **紅燈** | 問題太多或規則不清楚 | 故事太大，需拆分或安排更多討論 |

**健康指標：**

| 指標 | 健康範圍 | 警告信號 |
|------|---------|---------|
| 工作坊時間 | ≤ 25 分鐘 | > 30 分鐘（故事太大） |
| 業務規則數 | 2–5 條 | > 8 條（需要拆分） |
| 具體範例數 | 5–15 個 | < 3 個（理解不足） |
| 待解決問題 | 0–3 個 | > 6 個（需求不清楚）|

---

## Clarify 循環（消除黃燈）

當準備度為 🟡 黃燈時，針對每個待解決問題進行釐清：

```
每輪 Clarify（固定六步）：
1. 問一題（選擇題優先，含 Others 選項）
2. 用戶回答
3. 更新文件（僅更新與該回答相關的部分）
4. 展示更新內容
5. 詢問：「這樣的更新是否符合預期？(y/n)」
6. y → 下一題　n → 詢問如何調整
```

**問題處理結果：**

| 結果 | 文件更新 |
|------|---------|
| 問題已解決 | 轉為新的業務規則 + 範例表格 |
| 問題非阻塞擱置 | 原問題加 `[非阻塞]` 標籤 |
| 問題導致拆分 | 暫停 Clarify，回到故事拆分 |

## Discovery vs. Formulation：關鍵區別

| | Discovery（Example Mapping） | Formulation（Gherkin） |
|---|---|---|
| **語言** | 自然語言、表格 | Gherkin（Given/When/Then） |
| **目的** | 探索、對齊理解 | 定義可執行規格 |
| **參與者** | Three Amigos + 業務人員 | 開發 + 測試 |
| **輸出** | `specs/*-discovery.md` | `features/*.feature` |
| **時間** | 開發前 | Discovery 完成後 |

**⚠️ Discovery 階段禁止使用 Given/When/Then**：這個階段的目的是促進討論，不是寫測試。過早進入 Gherkin 格式會限制思考，遺漏重要的業務規則。

## 從 Example Mapping 到 Gherkin 的轉換

```
Discovery 輸出                    Formulation 輸出
━━━━━━━━━━━━━━━━━━━━━━            ━━━━━━━━━━━━━━━━━━━━━━

📋 使用者故事              →      Feature: 描述
📏 業務規則                →      Rule: 或 Scenario: 名稱
📝 具體範例（表格每行）    →      Given-When-Then 步驟
   多個相似範例            →      Scenario Outline + Examples:
❓ 待解決問題              →      # TODO 或 @pending 標籤
```

## 完整範例：帳戶間轉帳

### Example Mapping 輸出（Discovery）

**Story：帳戶間轉帳**
As a 銀行客戶，I want 在帳戶間轉移資金，So that 我可以靈活管理財務。

**Acceptance Criteria：**
- [ ] 可以從支票帳戶轉帳到儲蓄帳戶
- [ ] 餘額不足時顯示明確錯誤訊息
- [ ] 單筆金額不得超過系統限額
- [ ] 轉帳失敗時帳戶餘額不變

**規則 1：餘額驗證**

| 範例 | 初始狀態 | 執行動作 | 預期結果 |
|------|---------|---------|----------|
| 1.1 - 餘額充足 | 支票 $1,000 | 轉 $200 | 支票 $800，成功 |
| 1.2 - 餘額不足 | 支票 $100  | 轉 $200 | 失敗「餘額不足」 |
| 1.3 - 恰好等於 | 支票 $200  | 轉 $200 | 支票 $0，成功 |

**規則 2：轉帳限額（單筆 ≤ $10,000）**

| 範例 | 初始狀態 | 執行動作 | 預期結果 |
|------|---------|---------|----------|
| 2.1 - 在限額內 | 支票 $20,000 | 轉 $9,999  | 成功 |
| 2.2 - 剛好達限 | 支票 $20,000 | 轉 $10,000 | 成功 |
| 2.3 - 超過限額 | 支票 $20,000 | 轉 $10,001 | 失敗「超過限額」 |

**待解決問題：**
1. VIP 客戶限額是否不同（$50,000）？ - 負責人：產品經理
2. 轉帳是即時到帳還是 T+1？ - 負責人：技術架構師

**評估：** 🟡 黃燈（需解決問題 1、2 後再開發）

### 轉換後 Gherkin（Formulation）

```gherkin
Feature: 銀行帳戶間轉帳

  Rule: 來源帳戶必須有足夠餘額

    Scenario Outline: 依餘額情況決定轉帳結果
      Given Tess 的支票帳戶有 $<初始餘額>
      When 她轉 $<金額> 到儲蓄帳戶
      Then 轉帳應該 <結果>

      Examples:
        | 情境     | 初始餘額 | 金額 | 結果              |
        | 餘額充足 | 1000     | 200  | 成功              |
        | 餘額不足 | 100      | 200  | 失敗並顯示餘額不足 |
        | 恰好等於 | 200      | 200  | 成功              |

  Rule: 單筆轉帳不得超過 $10,000

    Scenario Outline: 依限額情況決定轉帳結果
      Given Tess 的支票帳戶有 $20,000
      When 她轉 $<金額> 到儲蓄帳戶
      Then 轉帳應該 <結果>

      Examples:
        | 情境     | 金額  | 結果              |
        | 在限額內 | 9999  | 成功              |
        | 剛好達限 | 10000 | 成功              |
        | 超過限額 | 10001 | 失敗並顯示超過限額 |
```

# BDD 工作流程發現技能

## 概述

此技能引導使用者完成 BDD 的前期工作流程：

1. **功能描述階段**：協助撰寫清晰的使用者故事
2. **故事評估與拆分階段**：評估故事大小，必要時應用拆分模式
3. **Example Mapping 階段**：為每個小故事發現業務規則和具體測試範例

**⚠️ 重要：Discovery 階段使用自然語言和表格**
- ✅ 使用表格或簡單自然語言描述範例
- ❌ **不使用 Given-When-Then 格式**（那是 Formulation 階段的事）
- 目的是促進討論和探索，不是寫測試

## 核心概念

### 📐 功能的組織層次

BDD 使用範例來描述與組織功能，並按以下層次結構思考：

| 層次 | 定義 | 說明 |
|------|------|------|
| **能力（Capability）** | 支持某個業務目標的能力 | 與具體實作方式無關，例如「讓客戶能夠管理資金」 |
| **功能（Feature）** | 可交付的軟體功能，為使用者提供某種能力 | 例如「帳戶間轉帳」、「查看交易記錄」 |
| **使用者故事（User Story）** | 規劃與交付功能的單位 | 敏捷專案用於排程和開發，每個 Feature 可能有多個 Story |

**拆分原則：** 較大的 Feature 可以拆解為較小的 Feature，使其更容易組織與交付。拆分時遵循垂直切分（每塊都有完整業務價值），而非水平切分（技術層次分離）。

## 核心理念

### 🔍 刻意探索（Deliberate Discovery）

> 「在任何軟體專案中，最大的風險之一來自於自身的無知，因此應積極識別並降低不確定性。」

**刻意探索**是指主動、有意識地在開發前揭露未知，而不是等到問題在開發中或上線後才浮現。

這是 Discovery 階段存在的根本原因：

| 被動等待 | 刻意探索 |
|---------|---------|
| 等問題出現才處理 | 主動舉辦 Example Mapping 挖掘假設 |
| 開發到一半才發現需求不清楚 | 開發前就找出所有疑問（紅色便利貼） |
| 個人理解，各自為政 | Three Amigos 協作，建立共識 |
| 缺陷在測試階段才被發現 | 需求模糊在 Discovery 就被識別 |

**實踐方式：**
- 每個故事開發前，刻意安排 Example Mapping 會議
- 主動提問「什麼情況會失敗？」「邊界在哪？」「有例外嗎？」
- 把發現的未知記錄為問題（❓），指定負責人解決
- 發現問題越早，修復成本越低

### ⏳ 實際選擇（Real Options）

> 「在獲得足夠資訊並確認解決方案是最合適的之前，不應該過早承諾特定的解決方案。」

**實際選擇**是指將決策延遲到「最後負責任的時刻（Last Responsible Moment）」，而不是過早承諾一個可能錯誤的方向。

在 Discovery 階段的應用：

| 過早承諾 | 保留選擇 |
|---------|---------|
| 一開始就決定所有實作細節 | 只釐清業務行為，技術細節留給 Automation 階段 |
| 強迫解決所有問題才開始 | 識別「阻塞性問題」vs「可並行解決的問題」 |
| 大故事一次全部規劃 | 拆分小故事，逐一深入（DFS） |
| 現在就決定 VIP 規則 | 記錄為問題，等有更多資訊再決定 |

**判斷「最後負責任的時刻」：**
- ❓ **這個決定現在不做，會阻塞開發嗎？** → 是 → 現在解決
- ❓ **這個決定現在做，之後很難改嗎？** → 是 → 延遲決定
- ❓ **我們現在有足夠資訊做出好決定嗎？** → 否 → 先記錄，繼續探索

**在準備度評估中的體現：**
- 🟢 綠燈 = 已有足夠資訊，可以承諾開始開發
- 🟡 黃燈 = 尚有選擇空間，先解決關鍵問題
- 🔴 紅燈 = 太早承諾，需要更多探索

## 工作流程

### 階段一：功能描述

基於 `docs/chapter03/04-feature-description.md` 的指導原則，協助使用者：

1. **識別利益相關者**
   - 誰會使用這個功能？
   - 誰會從中受益？

2. **定義功能需求**
   - 具體需要什麼功能？
   - 功能的範圍是什麼？

3. **闡明業務價值**
   - 為什麼需要這個功能？
   - 解決什麼問題？
   - 創造什麼價值？

4. **選擇故事格式**
   - 格式一（目標導向）：`In order to... As a... I want...`
   - 格式二（角色導向）：`As a... I want... So that...`

5. **撰寫 Acceptance Criteria（驗收標準）**
   - 定義「完成」的標準
   - 使用可驗證的陳述
   - 涵蓋正常流程和異常情況
   - 作為 Example Mapping 的起點

6. **驗證 INVEST 原則**
   - Independent（獨立的）
   - Negotiable（可協商的）
   - Valuable（有價值的）
   - Estimable（可估算的）
   - Small（小的）
   - Testable（可測試的）

### 階段二：故事評估與拆分

基於 `docs/chapter03/06-story-splitting.md` 的指導原則，評估並拆分故事：

1. **評估故事大小**
   - 檢查是否符合 INVEST 的 S（Small）原則
   - 識別故事太大的徵兆：
     - 無法估算工作量
     - 超過一個 Sprint
     - Example Mapping 會超過 30 分鐘
     - 規則數量可能 > 8 個

2. **選擇拆分模式**（如果需要拆分）
   - 工作流程步驟（Workflow Steps）
   - 業務規則變化（Business Rule Variations）
   - 簡單/複雜（Simple/Complex）
   - 資料變化（Variations in Data）
   - CRUD 操作（Create/Read/Update/Delete）
   - 介面（Interface）
   - 效能/規模（Performance/Scale）

3. **應用拆分模式**
   - 確保垂直切分（每個故事包含完整功能切片）
   - 每個小故事都有獨立的業務價值
   - 避免水平切分（技術層次分離）

4. **驗證拆分結果**
   - 每個小故事符合 INVEST 原則
   - 可在 1-2 天內完成
   - 依賴關係最小化

5. **排序優先級**
   - 使用 MoSCoW 方法（Must/Should/Could/Won't）
   - 識別 MVP（最小可行產品）

### 階段三：Example Mapping

基於 `docs/chapter03/05-example-mapping.md` 的方法，引導使用者。
完整範例請參閱 [example-mapping.md](./example-mapping.md)。

1. **識別業務規則**（藍色便利貼 📏）
   - "在什麼情況下..."
   - "如果...那麼..."
   - "只有當..."

2. **創建具體範例**（綠色便利貼 📝）
   - 為每個規則提供至少 2 個範例
   - 涵蓋正常情況和異常情況
   - 使用表格或簡單自然語言描述
   - **注意：Discovery 階段不使用 Given-When-Then**

3. **記錄待解決問題**（紅色便利貼 ❓）
   - 業務規則不明確的地方
   - 技術可行性問題
   - 整合問題

4. **評估準備度**
   - 綠燈：準備好開發
   - 黃燈：需要更多工作
   - 紅燈：故事太大，需拆分

## 實現邏輯

### 步驟 1：收集功能描述

```
如果用戶沒有提供描述：
  詢問：「請描述你想要實現的功能」

接收用戶輸入後：
  1. 總結理解
  2. 確認理解正確
```

### 步驟 2：引導撰寫使用者故事

```
詢問以下問題：

1. 利益相關者（As a）
   Q: "誰會使用這個功能？"
   Q: "他們的角色是什麼？"

2. 功能需求（I want）
   Q: "具體需要什麼功能或能力？"
   Q: "功能的核心是什麼？"

3. 業務價值（So that）
   Q: "為什麼需要這個功能？"
   Q: "它解決了什麼問題？"
   Q: "創造了什麼價值？"

基於回答，提供兩種格式的使用者故事：
- 格式一（目標導向）
- 格式二（角色導向）

詢問用戶偏好哪種格式
```

### 步驟 3：驗證 INVEST 原則

```
檢查使用者故事是否符合 INVEST：

如果故事太大（S - Small）：
  → 進入步驟 3.5（故事拆分）

如果缺乏明確價值（V - Valuable）：
  引導重新思考業務價值

如果無法測試（T - Testable）：
  協助定義驗收標準
```

### 步驟 3.5：故事拆分流程（如果需要）

```
說明拆分必要性：
"這個故事看起來比較大，讓我們將它拆分成更小的故事，
這樣每個故事都可以在 1-2 天內完成。"

階段 3.5.1：識別拆分模式
--------------------------------------
詢問引導問題：

Q1: "這個功能有明確的工作流程步驟嗎？"
    → 適用「工作流程步驟」模式

Q2: "不同的業務規則或條件會導致不同行為嗎？"
    → 適用「業務規則變化」模式

Q3: "是否有簡單版本和複雜版本的區別？"
    → 適用「簡單/複雜」模式

Q4: "是否涉及不同類型的資料或介面？"
    → 適用「資料變化」或「介面」模式

Q5: "是否是標準的 CRUD 操作？"
    → 適用「CRUD 操作」模式

階段 3.5.2：應用拆分模式
--------------------------------------
根據選擇的模式，引導拆分：

工作流程步驟：
  "讓我們將這個流程分解成步驟：
   第一步是什麼？第二步？...
   每個步驟都可以是一個獨立的故事。"

業務規則變化：
  "有哪些不同的情況或類型？
   我們可以為每種情況創建一個故事。"

簡單/複雜：
  "什麼是最基本的版本（MVP）？
   然後我們可以逐步增加複雜度。"

確保垂直切分：
  "每個小故事都要包含完整的功能切片
   （資料庫 + API + 介面 + 測試）"

階段 3.5.3：排序優先級
--------------------------------------
使用 MoSCoW 方法：

Q: "哪些故事是 Must have（必須有的）？"
   → 標記為 Sprint 1 / MVP

Q: "哪些故事是 Should have（應該有的）？"
   → 標記為 Sprint 2

Q: "哪些故事是 Could have（可以有的）？"
   → 標記為 Sprint 3

Q: "哪些故事是 Won't have（暫不做的）？"
   → 記錄到 Backlog

階段 3.5.4：驗證拆分結果
--------------------------------------
對每個小故事檢查：

□ 獨立的（不依賴其他故事）
□ 有業務價值（可獨立展示）
□ 可估算的（1-2 天）
□ 可測試的（清晰驗收標準）

如果都符合 → 拆分成功
如果不符合 → 重新調整

輸出格式：
展示拆分後的故事列表：
```
📋 拆分後的故事

Story 1.1：[名稱]
As a [角色]
I want [需求]
So that [價值]
優先級：Must have
估算：2 天

Story 1.2：[名稱]
...
```
```

### 步驟 4：Example Mapping 會議（為每個小故事）

```
說明 Example Mapping 流程：
"現在我們將進行 Example Mapping 來發現業務規則和範例。
我會扮演主持人角色，引導你完成這個過程。"

階段 2.1：探索業務規則（10 分鐘）
--------------------------------------
詢問引導問題：

Q1: "在什麼情況下這個功能會成功？"
    → 記錄為規則（📏）

Q2: "有哪些限制或約束？"
    → 記錄為規則（📏）

Q3: "什麼時候會失敗？"
    → 記錄為規則（📏）

    → 可能需要更多規則

對每個規則：
  展示格式：📏 規則 X：[規則描述]
  詢問："這個規則總是適用嗎？有例外嗎？"

階段 2.2：創建具體範例（10 分鐘）
--------------------------------------
對每個規則：

Q: "請提供一個這個規則的具體例子"
   → 記錄為範例（📝）

引導使用表格或自然語言描述範例：

**方式 1：表格格式**
  | 初始狀態 | 動作 | 預期結果 |
  |---------|------|----------|
  | [條件]  | [操作] | [結果]  |

**方式 2：自然語言描述**
  - 初始狀態：[描述]
  - 執行動作：[描述]
  - 預期結果：[描述]

**重要：Discovery 階段使用自然語言，不使用 Given-When-Then 格式**

確保涵蓋：
  ✓ 正常情況（快樂路徑）
  ✓ 異常情況（錯誤處理）
  ✓ 邊界情況（極限值）

階段 2.3：識別問題（5 分鐘）
--------------------------------------
詢問：

Q1: "有哪些地方還不清楚？"
    → 記錄為問題（❓）

Q2: "需要與誰確認？"
    → 標記負責人

Q3: "有技術可行性的疑慮嗎？"
    → 記錄為技術問題

Q4: "需要與其他系統整合嗎？"
    → 記錄為整合問題
```

```
### 步驟 6：Clarify 循環（釐清問題）

當 discovery 文件的準備度為 🟡 黃燈時，執行此步驟逐一釐清待解決問題，將故事推進到 🟢 綠燈狀態。

#### 循環規則

| 規則 | 說明 |
|------|------|
| **進入條件** | Discovery 完成後準備度為 🟡 黃燈，或用戶執行 `clarify` 子命令 |
| **每輪處理** | 一次只問一題、澄清一件事，確認後才進入下一題 |
| **結束條件** | 所有問題皆為「已解決」或「非阻塞擱置」，且至少一個故事達到 🟢 綠燈 |
| **中止條件** | 問題導致故事需要重新拆分 → 回到步驟 3.5，完成後重新進入 Clarify |
| **禁止事項** | 不可跳過問題、不可一次問多題、不可自行假設答案 |

#### 每輪流程（固定六步）

```
1. 問一題（選擇題優先，允許 Others 補充說明）
2. 用戶回答
3. 更新文件（僅更新與該回答相關的部分）
4. 展示更新內容
5. 詢問：「這樣的更新是否符合預期？(y/n)」
6. y → 進入下一題　n → 詢問如何調整，修正後重新確認
```

#### 提問格式

**多選題（優先使用）：**
```
Q: <問題描述>
- A) <選項>
- B) <選項>
- C) <選項>
- D) 其他（請說明）
```

**簡答題（選項難以窮舉時使用）：**
```
Q: <問題描述>（請提供簡短答案，10 字內）
```

提問原則：
- 問題必須具體、可回答，避免開放式問題
- 一個問題只澄清一件事
- 選擇題選項需互斥，涵蓋主要可能性

#### 每輪標頭格式

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
❓ 問題 [N/總數]：[問題描述]
   故事：Story [X.X] - [故事名稱]
   負責人：[負責人]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 各處理結果的文件更新格式

**情況 A：問題已解決 → 轉為業務規則**

規則：答案必須轉為可驗證的規則，每條規則至少需 2 個範例（正常 + 異常或邊界）

文件更新格式（加入對應 Story 的 Example Mapping）：
```
**規則 X：[規則名稱]**

[規則描述]

📝 範例表格：

| 範例 | 初始狀態 | 執行動作 | 預期結果 |
|------|---------|---------|----------|
| X.1 - [正常情況] | [描述] | [描述] | [描述] |
| X.2 - [異常/邊界] | [描述] | [描述] | [描述] |
```

從 ❓ 待解決問題清單移除該問題。

**情況 B：問題擱置（非阻塞）**

規則：確認此問題不影響當前迭代的核心開發；擱置後若無其餘阻塞問題，故事可升為 🟢

文件更新格式（在原問題後附加）：
```
1. [原問題描述] - 負責人：[名字] `[非阻塞]` 理由：[為什麼不阻塞] 後續：[誰負責、何時解決]
```

**情況 C：需要拆分**

規則：當答案揭示故事範圍比預期大或引入新業務維度，暫停 Clarify，回到步驟 3.5

文件更新格式（在問題後附加說明）：
```
1. [原問題描述] `[待拆分]` 理由：[拆分原因]
```

### 文檔儲存

**目錄結構：**
```
項目根目錄/
└── specs/
    ├── [功能名稱]-discovery.md       # 完整功能發現文檔
    └── ...
```

**檔案命名規則：**
- 功能名稱使用小寫和連字號（kebab-case）
- 範例：`account-management-discovery.md`、`payment-processing-discovery.md`

**儲存邏輯：**
1. 確保 `specs/` 目錄存在（使用 `mkdir -p specs`）
2. 生成完整的 Markdown 文檔
3. 使用 Write 工具儲存文件到 `specs/[功能名稱]-discovery.md`
4. 向用戶確認文件路徑

**文檔層次結構：**
```
功能（Feature）                          ← 最高層次
  ├── 功能描述
  └── 使用者故事（User Stories）        ← 功能下的多個故事
      ├── Story 1.1
      │   ├── 故事描述
      │   └── Example Mapping 結果
      ├── Story 1.2
      │   ├── 故事描述
      │   └── Example Mapping 結果
      └── ...
```

**設計理念：**
- `specs/` 符合 BDD 的「Specification by Example」概念
- 單一文檔包含完整功能的所有故事（功能描述 + 故事拆分 + Example Mapping）
- 清楚區分「功能」和「使用者故事」的層次
- 便於追溯和審查整個需求發現過程
```

```markdown
## 功能描述

提供客戶完整的帳戶管理能力，包括帳戶間資金轉移、交易記錄查詢等功能。

**功能價值：**
- 讓客戶能夠自主管理資金
- 提供透明的交易記錄
- 確保資金安全

## 使用者故事

### Story 1.1：帳戶間轉帳

**As a** 銀行客戶
**I want** 在我的帳戶間轉移資金
**So that** 我可以靈活管理我的財務

**優先級：** Must have
**估算：** 2 天
**依賴：** 無

**Acceptance Criteria（驗收標準）：**
- [ ] 可以從支票帳戶轉帳到儲蓄帳戶
- [ ] 可以從儲蓄帳戶轉帳到支票帳戶
- [ ] 轉帳前會驗證來源帳戶餘額是否充足
- [ ] 餘額不足時會顯示明確的錯誤訊息
- [ ] 單筆轉帳金額不得超過系統限額
- [ ] 轉帳成功後兩個帳戶餘額會正確更新
- [ ] 轉帳失敗時帳戶餘額不會改變

#### Example Mapping 結果

##### 📏 業務規則

**規則 1：餘額驗證**

來源帳戶必須有足夠的餘額來完成轉帳。

📝 **範例表格：**

| 範例 | 初始狀態 | 執行動作 | 預期結果 |
|------|---------|---------|----------|
| 1.1 - 餘額充足 | 支票帳戶 $1,000<br>儲蓄帳戶 $500 | 從支票轉 $200 到儲蓄 | 支票帳戶 $800<br>儲蓄帳戶 $700 |
| 1.2 - 餘額不足 | 支票帳戶 $100<br>儲蓄帳戶 $500 | 嘗試從支票轉 $200 到儲蓄 | 轉帳失敗<br>顯示「餘額不足」錯誤 |

**規則 2：轉帳限額**

單筆轉帳不得超過 $10,000

📝 **範例表格：**

| 範例 | 初始狀態 | 執行動作 | 預期結果 |
|------|---------|---------|----------|
| 2.1 - 在限額內 | 支票帳戶 $20,000 | 轉 $9,999 到儲蓄 | 轉帳成功 |
| 2.2 - 超過限額 | 支票帳戶 $20,000 | 嘗試轉 $10,001 到儲蓄 | 轉帳失敗<br>顯示「超過單筆限額」錯誤 |

##### ❓ 待解決問題

1. VIP 客戶的限額是否不同？ - 負責人：產品經理
2. 轉帳是即時處理還是有延遲？ - 負責人：技術架構師
```

## 最佳實踐

### ✅ Do（應該做）

1. **遵循 DFS 原則** ⭐ → [完整說明](./dfs-principle.md)
   - 一次專注一個故事
   - 完整探索當前故事的所有規則和範例
   - 完成後再進入下一個故事
   - 從優先級最高的故事開始（Must have）

2. **保持互動**
   - 每次只問 1-2 個問題
   - 根據回答調整下一個問題
   - 總結理解並確認

3. **撰寫清晰的 Acceptance Criteria**
   - 使用可驗證的陳述（明確的「完成」標準）
   - 涵蓋正常流程和異常情況
   - 作為 Example Mapping 的起點
   - 避免過於技術化的描述

4. **使用具體範例**
   - 要求提供實際數據
   - 避免抽象描述
   - 使用表格或自然語言（不用 Given-When-Then）

6. **記錄所有內容**
   - 規則、範例、問題都要記錄
   - 使用清晰的格式
   - 視覺化組織（emoji）

7. **驗證理解**
   - 重複使用者的話
   - 確認理解正確
   - 澄清模糊之處

8. **評估準備度**
   - 根據規則、範例、問題數量評估
   - 提供明確的下一步建議
   - 識別是否需要拆分故事

### ❌ Don't（避免做）

1. **不要使用 BFS（廣度優先）** ⭐ → [DFS vs BFS 對比](./dfs-principle.md)
   - 不要同時探索多個故事
   - 不要淺層掃過所有規則
   - 不要等所有故事都討論完才開始開發
   - 遵循 DFS 原則：深入一個，完成一個

2. **不要一次問太多**
   - 避免長問卷
   - 分階段進行
   - 保持對話流暢

3. **不要假設**
   - 不要猜測業務規則
   - 總是確認理解
   - 記錄不確定的地方

4. **不要跳過範例**
   - 規則必須有範例支持
   - 至少 2 個範例（正常+異常）
   - 範例要具體

5. **不要忽略問題**
   - 所有問題都要記錄
   - 指定負責人
   - 設定解決期限

## 工作流程總覽

```
階段一：功能描述
└─> 撰寫初始使用者故事
    └─> 驗證 INVEST 原則

階段二：故事評估與拆分（如果需要）
└─> 識別拆分模式
    └─> 應用拆分模式
        └─> 排序優先級
            └─> 驗證拆分結果

階段三：Example Mapping（為每個小故事）
└─> 探索業務規則
    └─> 創建具體範例
        └─> 識別待解決問題
            └─> 評估準備度
                ├─> 🟢 綠燈 → 輸出文件，進入 Formulation
                └─> 🟡 黃燈 → 輸出文件，進入 Clarify

階段四：Clarify（如果有待解決問題）
└─> 逐一釐清待解決問題
    └─> 答案 → 轉為新規則 + 範例
        └─> 重新評估準備度
            └─> 🟢 全部綠燈 → 進入 Formulation
```

## 🎯 DFS 原則（深度優先探索）

**核心理念：一次專注一個故事，完整探索後再進入下一個**

DFS (Depth-First Search) 是此技能的核心工作原則，確保每個故事都經過完整探索後才進入下一個。

### 快速摘要

- ✅ **做法**：選擇一個故事 → 完整 Example Mapping → 準備度評估 → 進入下一個
- ❌ **避免**：同時討論多個故事 → 淺層掃過所有規則 → 大批次處理

### 為什麼重要？

| DFS (深度優先) | BFS (廣度優先) |
|----------------|----------------|
| 完整細節、可立即開發 | 淺層理解、需等所有完成 |
| 持續交付、早期發現問題 | 大批次交付、後期暴露風險 |
| 認知負荷低、聚焦討論 | 認知負荷高、信息分散 |

# 🎯 DFS 原則（深度優先探索）

## 核心理念

**一次專注一個故事，完整探索後再進入下一個**

## 為什麼使用 DFS？

### ❌ BFS (廣度優先) - 避免這樣做

```
1. 列出所有拆分後的故事
2. 為所有故事做淺層 Example Mapping
3. 一起討論所有規則

→ 結果：信息過載、細節遺失、無法立即開始開發
```

**問題：**
- 同時追蹤多個故事，認知負荷高
- 每個故事只有淺層理解
- 需要等所有故事都討論完才能開始開發
- 信息分散，容易遺漏重要細節

### ✅ DFS (深度優先) - 應該這樣做

```
1. 選擇第一個故事（通常是 MVP 中的 Story 1.1）
2. 深入進行完整的 Example Mapping
   - 探索所有業務規則
   - 創建所有相關範例
   - 識別所有待解決問題
3. 評估準備度
4. 如果準備好 → 可以立即開始開發
5. 重複步驟 1-4 處理下一個故事

→ 結果：每個故事都有完整細節、可以立即開發、持續交付價值
```

**優勢：**
- 一次只專注一個故事，認知負荷低
- 每個故事都有深入的理解和完整的範例
- 完成一個就可以立即開發，不需等待
- 持續交付價值，而非大批次交付

## DFS 工作流程

```
Story 1.1 (Must have)
│
├─ Example Mapping (深入)
│  ├─ 規則 1 → 範例 1.1, 1.2
│  ├─ 規則 2 → 範例 2.1, 2.2
│  └─ 問題識別 → 全部記錄
│
├─ 準備度評估
│  └─ 綠燈 ✅
│
└─ 可以開始開發！
    │
    ├─ 編寫 Gherkin (使用 /bdd-workflow-formulation)
    ├─ 實現步驟定義
    └─ 開發完成
        │
        └─ 進入下一個故事
            │
            Story 1.2 (Must have)
            │
            ├─ Example Mapping (深入)
            └─ ...
```

## 實踐指南

### 步驟 1：選擇當前故事
- 從優先級最高的開始（Must have）
- 通常是 Story 1.1
- 遵循 MVP 範圍

### 步驟 2：深入 Example Mapping
- **不要急於進入下一個故事**
- 充分探索所有規則和範例
- 確保涵蓋正常、異常、邊界情況
- 記錄所有疑問和問題

### 步驟 3：完成當前故事後再繼續
- 確保準備度為綠燈（或黃燈但問題已記錄）
- 問題已記錄並指派負責人
- 團隊對這個故事有共識
- **然後才進入下一個故事**

### 步驟 4：迭代進行
```
Story 1.1 完成 → 可以開始開發 → Story 1.2
Story 1.2 完成 → 可以開始開發 → Story 1.3
Story 1.3 完成 → 可以開始開發 → Story 1.4
```

每個故事都經過完整的 DFS 處理。

## 優勢對比

| 特性 | DFS (深度優先) | BFS (廣度優先) |
|------|----------------|----------------|
| **專注度** | 高 - 一次一個故事 | 低 - 同時處理多個 |
| **信息保留** | 完整 - 深入細節 | 部分 - 淺層概覽 |
| **可開發性** | 立即 - 完成即可開發 | 延遲 - 需等所有完成 |
| **風險控制** | 早期發現問題 | 後期集中暴露 |
| **交付節奏** | 持續交付 | 大批次交付 |
| **認知負荷** | 低 - 逐個處理 | 高 - 同時追蹤 |
| **團隊協作** | 聚焦討論 | 分散注意力 |
| **價值實現** | 快速 - 每個故事獨立 | 慢速 - 等待全部 |

## 範例對話

### ❌ 錯誤做法（BFS）

```
Claude: "我們已經拆分成 5 個故事，現在我們快速過一遍每個故事的業務規則..."

[對 Story 1.1] "好，這個故事有哪些規則？"
User: "規則 1 是..."
Claude: "好的，先記下來。"

[對 Story 1.2] "這個故事呢？"
User: "規則 2 是..."
Claude: "好的，也先記下來。"

[對 Story 1.3] "接下來這個..."

→ 結果：每個故事都只有淺層理解，沒有具體範例，無法立即開始開發
```

### ✅ 正確做法（DFS）

```
Claude: "我們已經拆分成 5 個故事。讓我們先專注在 Story 1.1（使用者登入）。"

Claude: "對於使用者登入，在什麼情況下會成功？"
User: "帳號和密碼正確時會成功。"
Claude: "好的，讓我們深入探索這個規則。請提供一個具體的例子。"
User: "Given 使用者 'john' 密碼是 'pass123'..."
Claude: "很好！那失敗的情況呢？"
User: "密碼錯誤時會失敗..."

[完整的 Example Mapping for Story 1.1]

Claude: "✅ Story 1.1 的 Example Mapping 已完成！"
Claude: "- 規則數量：3"
Claude: "- 範例數量：6"
Claude: "- 準備度：綠燈"
Claude: ""
Claude: "你可以開始開發 Story 1.1 了。使用 /bdd-workflow-formulation 生成 Gherkin。"
Claude: ""
Claude: "接下來要進行 Story 1.2 (使用者登出) 的 Example Mapping 嗎？"
```

## 與敏捷開發的契合

DFS 原則完美契合敏捷開發的核心價值觀：

### 1. 持續交付價值
```
❌ BFS：完成 5 個故事的淺層分析 → 等待全部深入分析 → 一次性開發
✅ DFS：完成 Story 1 → 立即開發 → 交付 → 完成 Story 2 → 立即開發 → 交付
```

### 2. 快速反饋循環
```
DFS 允許：
Story 1.1 完成 → 開發 → 測試 → 反饋
                           ↓
              用反饋改進 Story 1.2 的 Example Mapping
```

### 3. 降低風險
```
早期發現的問題（Story 1.1）可以影響後續故事的設計
而不是等到全部分析完才發現問題
```

### 4. 小批量工作
```
每次只處理一個故事 = 小批量
減少在製品（WIP）
提高流動效率
```

## 時間分配範例

### ❌ BFS 方式（總時間 3 小時）
```
故事列舉：15 分鐘
  ├─ Story 1.1, 1.2, 1.3, 1.4, 1.5

淺層 Mapping：1.5 小時
  ├─ Story 1.1 規則：18 分鐘
  ├─ Story 1.2 規則：18 分鐘
  ├─ Story 1.3 規則：18 分鐘
  ├─ Story 1.4 規則：18 分鐘
  └─ Story 1.5 規則：18 分鐘

回過頭深入範例：1.25 小時
  ├─ Story 1.1 範例：15 分鐘
  ├─ Story 1.2 範例：15 分鐘
  ├─ ...（重複切換上下文）

問題：信息分散、多次切換上下文、認知負荷高
```

### ✅ DFS 方式（總時間 3 小時）
```
Session 1: Story 1.1 (36 分鐘)
  ├─ 規則探索：12 分鐘
  ├─ 範例創建：18 分鐘
  └─ 問題識別：6 分鐘
  ✅ 準備度：綠燈 → 可立即開發

Session 2: Story 1.2 (36 分鐘)
  ├─ 規則探索：12 分鐘
  ├─ 範例創建：18 分鐘
  └─ 問題識別：6 分鐘
  ✅ 準備度：綠燈 → 可立即開發

Session 3-5: Story 1.3, 1.4, 1.5
  (...同樣模式...)

優勢：
- 每個 session 有明確成果
- 可以分散在不同時間進行
- Story 1.1 完成後可立即開始開發
- 不需要等待全部完成
```

## 何時可以例外？

DFS 是預設原則，但以下情況可以考慮調整：

### 情況 1：故事高度相關
```
如果 Story 1.2 完全依賴 Story 1.1 的規則：
→ 可以在討論 Story 1.1 時，同時記錄 Story 1.2 的衍生規則
→ 但仍要完整完成 Story 1.1 後，再正式處理 Story 1.2
```

### 情況 2：快速驗證可行性
```
如果對整體方向有疑慮：
→ 可以對所有故事做 5 分鐘快速掃描（識別紅旗）
→ 然後立即回到 DFS，深入第一個故事
```

### 情況 3：團隊資源分配
```
如果有多個團隊平行開發：
→ 每個團隊仍然遵循 DFS
→ Team A: Story 1.1 → Story 1.2
→ Team B: Story 2.1 → Story 2.2
```

## 檢查清單

在進入下一個故事之前，確認當前故事：

- [ ] 所有業務規則都已識別
- [ ] 每個規則至少有 2 個範例（正常 + 異常）
- [ ] 所有範例都使用 Given-When-Then 格式
- [ ] 所有問題都已記錄並指派負責人
- [ ] 準備度評估已完成（綠燈/黃燈/紅燈）
- [ ] 團隊對這個故事有共識
- [ ] 可以立即開始開發（或問題已有解決計畫）

## 總結

**DFS 的本質：**
> 與其對 10 個故事有淺層理解，不如對 1 個故事有深入理解並立即開始創造價值。

**記住：**
- 🎯 一次一個故事
- 🔍 深入完整探索
- ✅ 完成再繼續
- 🚀 持續交付價值

**口訣：**
```
深入一個，完成一個
不貪多，求精準
持續交付，快速反饋
```
